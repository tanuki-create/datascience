7. **プロの扱い方** — 設計・運用・計測のチェックリスト

## エッジ層の責務スタック（1枚図）

**エッジ層**（ユーザーに地理的・ネットワーク的に近い入口側の層）において、クライアントからバックエンド（サーバー側の業務処理層）へ向かうと、しばしば **リバースプロキシ**（クライアントはプロキシだけを見る逆方向プロキシ）**→（L7）ロードバランサー／API Gateway** のように層が積まれます。**L4**（OSI第4層・Transport層：TCP/UDPの接続先などで振り分け、ペイロードは原則見ない）と **L7**（第7層・Application層：HTTPのURLやヘッダなど**内容**で振り分け可能）のどちらで切るか、**TLS終端**（HTTPSの復号をここで行い、内側は別ポリシー）と認証をどこで握るかが論点です。

```mermaid
flowchart LR
  Client[Client]
  RP[ReverseProxy]
  LB[LoadBalancer]
  GW[APIGateway]
  Svc[Services]

  Client --> RP
  RP --> LB
  LB --> GW
  GW --> Svc
```

## クイック対照表（よく混同されるペア）

| ペア | 一言の差 |
|------|-----------|
| Load Balancer（負荷分散装置） / Reverse Proxy（逆方向プロキシ） | LBは主に**分散・可用性**、RPは**TLS終端・URLルーティング・キャッシュ**などクライアント向けの入口制御 |
| API Gateway（APIの単一入口） / Reverse Proxy | GWは**API横断の契約**（認証・課金・バージョン・集約）まで含むことが多い。RPはより汎用の「裏向き代理」 |
| Rate limiting（レート制限） / Throttling（流量制御・絞り込み） | RLはしばしば**拒否（HTTP 429 Too Many Requests 等）**、Throttleは**処理速度の制限・キュー遅延**のニュアンス（用語は文脈依存） |
| Rate limiting / Backpressure（バックプレッシャー：下流に合わせて上流を抑える圧力） | RLは**入口の拒否/枠**、BPは**下流が上流に「流れを減らせ」と伝える**圧力制御 |
| JWT（JSON Web Token：署名付きトークン形式） / OAuth（認可の標準フロー） | JWTは**トークンの表現形式**、OAuthは**認可フロー（委譲）の枠組み**（JWTをOAuthで使うことは多い） |
| GraphQL / gRPC（Google製RPC、HTTP/2+Protocol Buffers） | GraphQLは主に**クライアント主導のデータ取得（HTTP上）**、gRPCは**IDL（インタフェース定義言語）駆動のRPC（HTTP/2）** |
| CAP（分散環境の一貫性・可用性・分断耐性のトレードオフ） / ACID（DBトランザクションの性質） | CAPは**分散レプリカ間の一貫性と可用性のトレードオフ**、ACIDは**主に単一DBトランザクション**の性質 |
| Circuit breaker（サーキットブレーカー：障害伝播を止める） / Idempotency（べき等性） | CBは**障害の伝播停止**、べき等性は**同じ操作の重複を安全にする** |
| Deadlock（デッドロック） / Livelock（ライブロック） | デッドロックは**待ちの循環で停止**、ライブロックは**進捗はあるが実質前に進まない** |
| Cache stampede（キャッシュスタンピード） / Cache invalidation（キャッシュ無効化） | スタンピードは**大量同時ミス**の問題、無効化は**いつ鮮度を捨てるか**の設計 |

## トラフィック・エッジ

### API Gateway

- **現場のプロの解説**  
  「単なる**ルータ**（転送ルール）」ではなく、**外部公開APIの契約と運用の境界**として使われることが多い。**マイクロサービス**（小さく分割したバックエンド群）化が進むほど、認証・**レート制限**（一定時間あたりの呼び出し上限）・**バージョニング**（API版の互換管理）・**観測**（**相関ID**（Correlation ID：リクエストを跨いで追跡するID））の集約点として重みが増える。

- **実際のプロダクトでの役割**  
  クライアントをバックエンドの**トポロジ**（Topology：構成の繋がり）から切り離し、**BFF**（Backend for Frontend：フロント向けに集約するAPI層）的な集約や **APIキー**（呼び出し側識別子）**／課金単位**、**WAF**（Web Application Firewall：アプリ層の攻撃を遮断する壁）連携の手前を担う。

- **似た概念の違い**  
  **Reverse Proxy** より API **ライフサイクル**（公開・廃止・互換の運用）に踏み込む。**Load Balancer** は多くの場合L4/L7の転送・**健全性**（Health：ヘルスチェック）が主眼で、ドメインのAPIポリシーはGW側に寄せることが多い。

- **なぜ本当に必要か**  
  クライアントがバックエンドの **数・配置・認証方式** まで知る必要があると、**変更のたびにアプリを全更新**し、セキュリティ境界もばらける。**公開APIの契約**（誰が何を提供し、どう課金・版管理するか）を **1つの面** に集約するため。

- **ないと何が困るか**  
  サービスごとにURL・認証・キー運用が分散し、**観測（ログ・メトリクス）が追えない**、**課金やレート制限がチームごとにバラバラ**、インシデント時に「どの入口の責務か」が争点になる。

- **歴史・背景と広まり方**  
  **SOA**（Service-Oriented Architecture）や **ESB**（Enterprise Service Bus）で「単一入口」の思想は以前からあったが、**クラウドのマネージドAPI**（AWS API Gateway 2015年頃〜等）と **マイクロサービス** の普及で「外部向けAPIの契約・課金・バージョンを一箇所に」というニーズが顕在化した。オープンソースでは **Kong**、**Tyk**、**Envoy** を入口に据える構成が広まり、**Kubernetes Ingress / Gateway API** と統合される形が主流になっている。

- **プロの扱い方**  
  - 認証の実体（検証鍵の**ローテ**（ローテーション：定期差し替え）、**JWKS**（JSON Web Key Set：公開鍵の一覧を配る仕組み）取得失敗時の挙動）を明文化する  
  - タイムアウト階層（クライアント→GW→**下流**（下位のサービス））を二重にしない  
  - 相関ID・**サンプリング**（ログの一部だけを残す割り込み）方針を統一する  

- **プロになるための質問**  
  - このGWは「公開APIの契約」として何を唯一の正とするか（ルーティング・認証・課金・バージョンのどこまでを担うか）。  
  - 下流1サービスが遅いとき、GWはタイムアウト・**リトライ**（再試行）・**サーキット**（サーキットブレーカー）をどう配分し、ユーザーに何を返すか。  
  - **マルチリージョン**（複数リージョン）時、GW手前の**DNS**（名前解決）と**TLS証明書**（HTTPSの身分証明）・WAFの責務境界は誰が持つか。  

**詳細:** [API Gateway パターン](./17_common_patterns/api_gateway.md)

### Load Balancer

- **現場のプロの解説**  
  **トラフィック**（通信量・リクエストの流れ）を複数**ターゲット**（転送先）に分散し、**ヘルスチェック**（生存・業務可否の確認）で不調ノードを外す。L4（TCP/UDP）とL7（HTTP）でできることが変わる。

- **実際のプロダクトでの役割**  
  **可用性**（サービスが継続して使える度合い）向上、**ローリングデプロイ**（段階的に新旧を入れ替える）時の切り替え、**ゾーン**（データセンタ内の冗長区画）跨ぎの均等化。**Kubernetes**（コンテナオーケストレーション）では **Service** / **Ingress**（外部HTTPルーティング）配下の実体として現れることが多い。

- **似た概念の違い**  
  **Reverse Proxy** は1台の「代理」としての振る舞い全般を指すこともあり、LBは**複数バックエンド**（裏側のサーバ群）への分散が主目的。**API Gateway** はAPIポリシー寄り。

- **なぜ本当に必要か**  
  **1台サーバ＝単一障害点**であり、トラフィック増に **縦スケールだけ** では限界がある。**複数台で処理能力と冗長**を得るには、手前で **振り分けと不健康ノードの除外** が要るから。

- **ないと何が困るか**  
  1台落ちれば **全停止**、負荷が1台に集中して **遅延と障害** が一体になる。デプロイで **無停止切替** もしづらい。

- **歴史・背景と広まり方**  
  インターネット普及期の **DNS ラウンドロビン** や **ハードウェアLB**（F5 等）から、Webバブル期〜2000年代に **L4/L7 アプライアンス** がデータセンタの定番となった。**クラウドの ELB/ALB**（AWS 2009年 ELB 以降）で「APIとしてのLB」が誰でも使えるようになり、**仮想化・コンテナ** と組み合わせて現在の形に収斂している。

- **プロの扱い方**  
  - **スティッキセッション**（同一クライアントを同じバックエンドへ固定）は最後の手段（**ステート**（状態）を共有ストアへ）  
  - ヘルスチェックは「プロセス生存」ではなく**業務的に意味のある readiness**（**Readiness**：受け付け準備完了）に寄せる  
  - **コネクション枯渇**（**keep-alive**（接続再利用）、タイムアウト）を監視する  

- **プロになるための質問**  
  - L4 と L7 のどちらで振り分けるか、そのとき **TLS 終端**（HTTPSの復号をどこで行うか）はどこか。  
  - デプロイ中に「古いバイナリ」と「新バイナリ」が混在するとき、LB はどう振る舞うべきか（**ドレイン**（既存接続の処理完了後に切離し）、**重み**（Weighted：配分比率））。  
  - ゾーン障害時、同じリージョン内の残りノードへの再配分が **SLO**（Service Level Objective：目標とする運用品質）を満たすか。  

**詳細:** [Load Balancing](./17_common_patterns/load_balancing.md)

### Reverse Proxy

- **現場のプロの解説**  
  クライアントはプロキシにだけ話し、プロキシが**バックエンドへ代理要求**する。**TLS終端**、静的配信、**パスルーティング**（URLパスで振り分け）、**Basic**（Basic認証）/WAF など**入口の横断関心**（複数サービス共通の関心事）を集約しやすい。

- **実際のプロダクトでの役割**  
  本番では Nginx / Envoy / Cloud LB の手前・併置など多様。**CDN**（Content Delivery Network：地理的に近い edge から配信）**エッジ**と役割が重なる部分もある（キャッシュ・TLS）。

- **似た概念の違い**  
  **Forward proxy**（**フォワードプロキシ**：クライアント側の出口／社内出口）とは逆方向。**API Gateway** より汎用語で、製品によっては同一レイヤに統合される。

- **なぜ本当に必要か**  
  **TLS・圧縮・キャッシュ・パス振り分け** をアプリプロセスから切り離し、**同じ入口IP・証明書**で複数バックエンドを扱うため。アプリは **ビジネスロジック** に集中できる。

- **ないと何が困るか**  
  各言語ランタイムで **証明書運用・HTTP/2・バッファ設定** を重複実装し、**設定ズレ**や **セキュリティパッチ** がばらつく。静的ファイルまでアプリが抱え **無駄なCPU** を使う。

- **歴史・背景と広まり方**  
  1990年代の **CERN プロキシ** や **Squid**（キャッシュ・共有出口）から、Webサーバの **リバースプロキシ**（Apache **mod_proxy**、**Nginx** 2000年代〜）として「公開IPは1つ、裏に複数アプリ」とするパターンが定着した。**TLS終端**・静的配信・**WAF** 前段としてのRPは、HTTPS全盛期とともに標準装備化した。

- **プロの扱い方**  
  - バッファ・**ボディサイズ制限**（巨大ボディによる**アップロード攻撃**）を設定する  
  - `X-Forwarded-*`（**X-Forwarded-For** 等：元クライアントIPを伝える拡張ヘッダ）の信頼境界（誰が付与してよいか）を定義する  
  - キャッシュ可能なGETと不可なGETのルールを分ける  

- **プロになるための質問**  
  - 信頼できるプロキシチェーンの先頭はどこか、偽の `X-Forwarded-For` をどう弾くか。  
  - **WebSocket**（双方向の長寿命HTTP）や gRPC **長接続**（コネクションを張りっぱなしにする）とき、RP のタイムアウトと**アイドル切断**（無通信時の切断）はどう設計するか。  
  - 静的キャッシュと認証付きAPIを同じホストで配るとき、**キャッシュキー**と**認可**（Authorization：誰が何をしてよいか）の衝突をどう防ぐか。

### Rate Limiting

- **現場のプロの解説**  
  単位時間あたりの要求数・**帯域**（データ転送量）を**上限で打ち切る**考え方。スパム・誤実装のループ・共有リソースの浪費を防ぐ**最も外側のフェンス**（防衛線）になりやすい。

- **実際のプロダクトでの役割**  
  公開API、ログイン試行、高コスト**エンドポイント**（URL先の処理単位）の保護。**テナント**（SaaSの顧客単位）**／ユーザー／IP／APIキー**などのキー設計が本質。

- **似た概念の違い**  
  **Throttling** は「速度を抑える」全般を指し、実装としてはキュー遅延・**トークンバケット**（一定レートでトークンが溜まるアルゴリズム）等。**Backpressure** は下流からの圧力で、責務がシステム内部に寄る。

- **なぜ本当に必要か**  
  **無限に速いクライアント**と**有限のサーバ／DB**の間に、**公平性と生存**のフェンスが要る。悪意・バグ・バーストを **入口で数として扱う**ため。

- **ないと何が困るか**  
  1クライアントやバグループが **全リソースを食いつぶし**、他テナントや正常ユーザが巻き添え。**コスト（従量課金）暴走**、**DDoSに近い自傷**、障害時の **原因特定が困難**。

- **歴史・背景と広まり方**  
  電話網の **トラフィック制御** や、**トークンバケット／リーキーバケット**（1980年代〜ネットワーク理論）に起源がある。HTTP API 全盛期に **公開APIの乱用防止**・**課金単位** としてクラウド（**AWS API Gateway** のスロットリング等）や **API管理製品** に組み込まれ、モバイルの再送・スパム対策でも標準化した。

- **プロの扱い方**  
  - HTTP **429** + `Retry-After`（再試行まで待つ秒数）または明確なエラーコード方針  
  - 分散環境では**中央集権**（Redis等の共有ストア）か**ローカル＋許容誤差**（各ノードで近似）かを決める  
  - 誤って自分のクライアントを締め出していないか（モバイル**再送**）を検証する  

- **プロになるための質問**  
  - レート制限のキーは「IP」「ユーザー」「APIキー」のどれを主にし、悪意ある分散をどう扱うか。  
  - **バースト**（一瞬のまとめ送り）と**スライディングウィンドウ**（時間窓を滑らせて数える方式）、どちらを採用し**メトリクス**（監視指標）でどう検証するか。  
  - **レプリカ**（複製サーバ）数が増えたとき、グローバルカウンタなしで「おおむね」制限する設計は許容できるか。  

**詳細:** [Rate Limiting](./17_common_patterns/rate_limiting.md)

### Throttling

- **現場のプロの解説**  
  用語は文脈でブレるが、現場では **「拒否」より「ならす」**（**キュー**（待ち行列）に積む・トークンを遅らせる）ニュアンスで使われることが多い。レート制限とセットで語られる。

- **実際のプロダクトでの役割**  
  バックプレッシャーの手前で**平滑化**（ピークを削って平らにする）し、DBや外部APIの**スパイク**（急激な負荷の山）を削る。メール送信・**プッシュ通知**・**バッチワーカー**（非同期処理）向けキューとも相性が良い。

- **似た概念の違い**  
  **Rate limiting** が「枠」を強調するのに対し、**Throttling** は**処理の速度・同時実行**（**並行度**）を抑えるイメージ。**Backpressure** はより下流主導。

- **なぜ本当に必要か**  
  **即拒否（429）** ではビジネス上まずい処理（メール・通知・レポート）で、**遅延は許すが無限には積まない**という中間が要る。**尖った負荷を時間でならす**ため。

- **ないと何が困るか**  
  一瞬のスパイクで **DBや外部APIが飽和**、または **429だらけで成功率が落ちる**。優先度の低い処理が **高い処理を押しのける**。

- **歴史・背景と広まり方**  
  元々は **CPU／ネットワークの輻輳制御** と同じ発想で、**メッセージブローカー**（**AMQP** 等）やメール配信で「何通／秒」の **シェーピング** として使われてきた。クラウドの **マネージドキュー**・**サーバレス** の同時実行上限とセットで語られるようになり、**429 ではなく遅延でならす**運用も一般化した。

- **プロの扱い方**  
  - キュー滞留時間（SLO）と**ワーカー**（処理プロセス）数の見える化  
  - **優先度キュー**（優先度付き待ち行列）の是非を決める  
  - 「遅延する」ことを**ユーザー体験**（UX）として許容するか事前に合意する  

- **プロになるための質問**  
  - 即時429（拒否）とキュー待ち（遅延）、ビジネス上どちらが許容か。  
  - 同時実行数を絞ると**スループット**（単位時間の処理量）が頭打ちになるとき、**水平スケール**（台数を増やす）とどう両立するか。  
  - スロットリング後の「**順序保証**」は必要か、不要ならどうドキュメントするか。

### Backpressure

- **現場のプロの解説**  
  **下流**（処理の後段）が処理できないとき、**上流**（前段）に「送るな／遅れよ」というシグナルを返す考え方。**ミドルウェア**（中間層ソフト）・**ストリーム**（逐次データ流）・**リアクティブ**（非同期・イベント駆動の背圧モデル）系で頻出。

- **実際のプロダクトでの役割**  
  メモリ内キューの無制限肥大を防ぎ、**OOM**（Out Of Memory：メモリ枯渇）より前に流量を制御する。Kafka の `max.in.flight`（未確認送信の上限）、gRPC の**フロー制御**、**Reactive Streams**（背圧付きストリームの仕様）等とセットで語られる。

- **似た概念の違い**  
  **Rate limiting** は入口の**ポリシー**（ルール）が中心。**Throttling** は速度調整全般。**Backpressure** は**下流の実処理能力**に合わせて上流を抑える点が強い。

- **なぜ本当に必要か**  
  **生産速度＞消費速度**が続くと、メモリ上のキューだけで **無限にバッファ** できず **OOM** や **遅延の爆発** が起きる。上流が **「まだ送るな」と知る**仕組みが要る。

- **ないと何が困るか**  
  **メモリ枯渇**、**GCストップ**、**p99 が読めない**ほどの待ち行列。カスケードで **全サービスが遅延** して原因特定が困難。

- **歴史・背景と広まり方**  
  **パイプやTCPのフロー制御**、製造ラインの **CONWIP** など「下流が詰まれば上流が止まる」モデルに相当する。**Reactive Streams**（2013年〜、**RxJava** 等）でプログラミング言語に名前が定着し、**Kafka** の消費速度・**gRPC** のフロー制御とセットで分散システムの常識になった。

- **プロの扱い方**  
  - キュー深さ・滞留件数を**アラート**（警告）にする  
  - **ブロッキング**（処理が止まって待つ）と**ドロップ**（捨てる）のどちらを選ぶか（**損失許容**か**遅延許容**か）  
  - サーキットブレーカーと併用し、**連鎖障害**（一つが連鎖的に全滅）を防ぐ  

- **プロになるための質問**  
  - 下流が遅いとき、上流は「ブロック」「バッファ」「ドロップ」のどれを選び、どのメトリクスで判断するか。  
  - ストリーム処理で「遅い**コンシューマ**」（Consumer：メッセージ消費側）を検知したら、**パーティション**（分割単位）再割当とどう組み合わせるか。  
  - バックプレッシャーを「ユーザーに見える遅延」に変換しないためのUI/UXはあるか。

## API 形態・連携

### GraphQL

- **現場のプロの解説**  
  クライアントが**欲しいフィールドを指定**して取得する。**過取得**（不要なデータまで取る）・**不足取得**（足りず追加リクエストが必要）を減らせる一方、**N+1**（関連1件ごとにクエリが増える問題）・**複雑度攻撃**（重いクエリで負荷をかける）が運用課題になる。

- **実際のプロダクトでの役割**  
  モバイル複数画面やBFFでAPI数を減らすのに強い。**スキーマ**（型とフィールドの宣言）変更の**互換性ポリシー**（**deprecated**（非推奨マーク）、**breaking change**（互換破壊の変更））が長期的な成否を分ける。

- **似た概念の違い**  
  **gRPC** はIDL・**RPC**（Remote Procedure Call：遠隔手続き呼び出し）・バイナリ志向。**REST** はリソース指向のHTTP。**GraphQL** は単一エンドポイント上の**クエリ言語**という役割分担。

- **なぜ本当に必要か**  
  **画面ごとにRESTを増やす**と **エンドポイント爆発** と **過取得／不足取得**（ラウンドトリップ増）が起きる。**1リクエストで必要形状だけ**取れる表現力が、モバイルやBFFでコストとUXの両立に効くから。

- **ないと何が困るか**  
  **無駄なJSON**、**N+1 のREST版**（細かいAPIを連打）、**クライアントとサーバの結合が強い**（フィールド追加のたびに両方更新）。ただし **運用（複雑度制限・認可）** を怠ると逆に危険。

- **歴史・背景と広まり方**  
  **Facebook が2012年頃にモバイル向けに開発**し、**2015年に仕様を公開**。「RESTだとエンドポイント爆発」「過取得／不足取得」への反動として、**クライアント主導のデータ取得**が支持された。**Apollo**、**Relay** などのクライアント／サーバ実装とともに広がり、**BFF** パターンとも相性がよい。

- **プロの扱い方**  
  - クエリ深度・コスト上限（**complexity limit**：クエリの重さ上限）  
  - **DataLoader**（バッチ読み込み用ライブラリ）等で**バッチ化**・キャッシュ  
  - **認可**（Authorization）は**リゾルバ**（フィールド解決関数）**単位**で抜けが出やすいのでテストを厚くする  

- **プロになるための質問**  
  - 1リクエストがバックエンドに与える負荷を、**クエリコスト**でどう数式化するか。  
  - 「フィールド単位の認可」をどうテストし、一覧APIと詳細APIで抜けを防ぐか。  
  - N+1 を潰した後も、キャッシュと組み合わせたときの無効化はどう設計するか。  

---

### gRPC

- **現場のプロの解説**  
  **Protocol Buffers**（protobuf：バイナリのIDL）+ **HTTP/2**（多重化・ヘッダ圧縮等）ベースの **RPC**。**ストリーミング**（逐次送受信）、強い型、**コード生成**が利点。ブラウザ直は手間なので **grpc-web**（ブラウザ向けゲートウェイ経由）やゲートウェイが挟まることが多い。

- **実際のプロダクトでの役割**  
  マイクロサービス間通信、高スループット内部API。外部公開は REST/**OpenAPI**（HTTP APIの仕様記述形式）と併用されることも多い。

- **似た概念の違い**  
  **GraphQL** は柔軟な読み取り、**gRPC** は契約固定の手続き呼び出し。**Webhooks** はサーバーからのプッシュ通知型の連携。

- **なぜ本当に必要か**  
  サービス間で **JSON REST** だけだと **スキーマが実装とドキュメントに分散**し、**バージョン破壊**が起きやすい。**IDL＋バイナリ＋HTTP/2** で **契約・性能・ストリーミング** を一度に扱うため（主に内部向け）。

- **ないと何が困るか**  
  **高QPS** でJSONパースと巨大ペイロードがボトルネック、**フィールドの意味が口頭頼み**、**双方向ストリーミング**が素朴なHTTPでは面倒。

- **歴史・背景と広まり方**  
  Google 内部の **Stubby** を一般化する目的で **2015年にオープンソース化**。**HTTP/2**（多重化・ヘッダ圧縮）と **Protocol Buffers**（**2008年に公開**されたバイナリIDL）を組み合わせ、**CNCF**（Cloud Native Computing Foundation）のプロジェクトとなり **Kubernetes・サービスメッシュ** 実装のデフォルト通信として普及した。

- **プロの扱い方**  
  - **デッドライン**（締切）・**キャンセル伝播**（**コンテキスト**（gRPCの文脈オブジェクト）経由で中止を伝播）  
  - バージョニング（**proto**（定義ファイル）の package/service の互換）  
  - ロードバランシング（L7 gRPC対応、**長寿命接続**）  

- **プロになるための質問**  
  - **ストリーミングRPC**で片側が落ちたとき、もう片側はどう検知・復旧するか。  
  - proto の**後方互換**変更ルール（フィールド番号、**optional**/**required**）をチームでどう守るか。  
  - ブラウザ公開が必要なとき、REST/GraphQL と gRPC の境界をどこに引くか。  

---

### Webhooks

- **現場のプロの解説**  
  イベント発生時に**相手のHTTPエンドポイントへPOST**（HTTP POSTで通知）する**プッシュ型**連携。**ポーリング**（定期的に問い合わせ）削減になるが、**配信保証**（必ず届くか）・**再送**・**順序**が課題。

- **実際のプロダクトでの役割**  
  **SaaS**（サービスとして提供するソフト）連携（決済完了、リポジトリイベント）。受信側は**署名検証**（**HMAC**（鍵付きハッシュによる改ざん検知））・**冪等処理**（何度来ても壊れない処理）が必須。

- **似た概念の違い**  
  **gRPC streaming** は常時接続の双方向。**メッセージキュー**（**ブローカー**（仲介サーバ）経由の**耐久化**（永続保存））が主。**Webhook** は「**HTTPコールバック**」（相手が指定したURLを叩く）という運用シンプルさが売り。

- **なぜ本当に必要か**  
  相手側が **常にポーリング** すると **レイテンシ・コスト・レート制限** が悪化する。**イベント駆動**で「起きたときだけ通知」し、統合を疎結合にするため。

- **ないと何が困るか**  
  **遅い反映**（ポーリング間隔）、**無駄なAPIコール**、**スケールするとポーリング地獄**。ただし Webhook だけでは **順序・再送・到達保証** は別設計が要る。

- **歴史・背景と広まり方**  
  **REST のポーリング負荷**への対案として「イベントが起きたらURLを叩く」が **2000年代の Webhooks** ブーム（**ProgrammableWeb** 等）で広まった。**GitHub・Stripe・Slack** などが「署名付きHTTPコールバック」を標準化し、**サーバレス**（Lambda + API Gateway）と組み合わせて統合のデフォルトになった。

- **プロの扱い方**  
  - 署名**シークレット**（共有秘密鍵）のローテ、タイムスタンプの許容**スキュー**（時刻ズレ）  
  - 再送で同イベントが何度も来る前提で**べき等**にする  
  - 受信側が落ちたときの**バックオフ**（再試行間隔を伸ばす）再送ポリシーを読む  

- **プロになるための質問**  
  - イベントの**順序**は保証されるか。保証されないなら受信側はどう整列するか。  
  - 配信遅延が「許容遅延」を超えたとき、ポーリングや手動**リカバリ**とどう併用するか。  
  - 受信URLをテナントごとに変える場合、署名鍵とURLの**バインド**（対応付け）をどう検証するか。

## 認証・信頼境界

### JWT

- **現場のプロの解説**  
  JSON**クレーム**（Claim：トークン内の主張・属性）を署名（または暗号化）した**自己記述的トークン**（中身だけで検証に必要な情報が揃う）。検証鍵が揃えば**ステートレス**（サーバがセッションを保持しない）**に近い検証**ができるが、失効・中間発行の扱いは設計が要る。

- **実際のプロダクトでの役割**  
  APIの**Bearerトークン**（`Authorization: Bearer ...` で送るトークン）、マイクロサービス間のユーザー文脈伝搬。**短命アクセストークン + リフレッシュトークン**（再発行用）が一般的。

- **似た概念の違い**  
  **OAuth** は認可フロー全体。**JWT** はその中で使われることもあれば独立して使われることもある。**mTLS** はトークンではなく**接続の身元**（TLSのクライアント証明書）。

- **なぜ本当に必要か**  
  各サービスが **セッションストアを共有** すると **スケールと障害域** がきつい。**署名付きクレーム**なら **検証鍵さえ配ればステートレス**に近づけ、**マイクロサービス間の文脈運搬**が単純になるから。

- **ないと何が困るか**  
  **Redis等への依存**、**データセンタ跨ぎの遅延**、**失効の即時性**とのトレードオフ。**トークン盗難**時の影響範囲を設計で抑えないと大事故。

- **歴史・背景と広まり方**  
  **JWT は2010年頃に提案され RFC 7519（2015）で標準化**。XML より軽い **JSON** で、**マイクロサービス間**の「署名付きクレーム運搬」に適した。**OAuth 2.0 のアクセストークン表現**や **OpenID Connect の ID トークン**として実装が広がり、**JWKS** による鍵配布とセットで定番になった。

- **プロの扱い方**  
  - `exp`（有効期限クレーム）短め、時刻スキュー許容  
  - アルゴリズム固定（`alg: none` 攻撃）・鍵ローテ（JWKS）  
  - 機密を**ペイロード**（本文）に入れない（署名は**改ざん防止**であり**暗号化**ではない）  

- **プロになるための質問**  
  - トークン失効を即時に反映したいとき、**ブラックリスト**と短い `exp` のどちらを選ぶか。  
  - `aud`（Audience：対象者） / `iss`（Issuer：発行者）を検証しないと何が起きるか、具体例を言えるか。  
  - マイクロサービス間でJWTを転送するとき、**誰が**署名検証し、**誰が**認可するか。  

---

### OAuth 2.0 / OpenID Connect

- **現場のプロの解説**  
  OAuthは**リソース所有者**（ユーザー本人）の代理で**クライアント**（アプリ）に権限を**委譲**する枠組み。**OIDC**（OpenID Connect）はその上に**IDトークン**（本人確認用）で**認証**を標準化。用語の混同（OAuth＝ログイン）に注意。

- **実際のプロダクトでの役割**  
  サードパーティ連携、社内**IdP**（Identity Provider：認証基盤）連携、モバイル/**SPA**（Single Page Application）での安全なトークン取得（**PKCE**（Proof Key for Code Exchange：公開クライアント向けのコード交換保護）等）。

- **似た概念の違い**  
  **JWT** は形式。**OAuth** はフローとロール。**SAML**（XMLベースのSSO標準）は別エコシステム（企業**SSO**（Single Sign-On）で多い）。

- **なぜ本当に必要か**  
  サードパーティアプリに **ユーザーのパスワードを渡さず**、**限定された権限（スコープ）だけ**を委譲するため。**監査可能な認可**と **ユーザーの同意** を両立する。

- **ないと何が困るか**  
  **クレデンシャル共有**、**権限の取り消し不能**、**パスワード流出の影響範囲が無限**。社内では **IdP なしのコピペ認証** が増える。

- **歴史・背景と広まり方**  
  **OAuth 1.0**（2007年頃）は署名地獄で複雑だったため、**IETF の OAuth 2.0**（**RFC 6749, 2012**）が「フローを絞った再設計」として普及。**ブラウザSPA・モバイルアプリ**の増加で **Authorization Code + PKCE** が主流に。**OpenID Connect**（2014年〜）が **OAuth 2 の上に認証層**を載せ、**Google/Microsoft 等の IdP** と一体で広まった。

- **プロの扱い方**  
  - **redirect URI**（認可後の戻り先URL）の厳密一致、**state**/**nonce**（CSRF・リプレイ対策用ランダム値）  
  - クライアント種別（**機密クライアント**（秘密鍵を持てる） vs **公開クライアント**（ブラウザ等））に応じたフロー選択  
  - **スコープ**（許可する操作の範囲）最小化と**監査ログ**  

- **プロになるための質問**  
  - **Authorization Code** + PKCE を選ぶ理由を、**Implicit**（暗黙フロー）や **ROPC**（Resource Owner Password Credentials：パスワード直渡し）と比較して説明できるか。  
  - **リフレッシュトークン**が漏れたときの影響範囲と、**ローテーション**（使うたびに入れ替え）方針は。  
  - 「ログイン」と「API連携」のスコープを分ける理由は何か。  

---

### mTLS（相互 TLS）

- **現場のプロの解説**  
  クライアントもサーバーも**証明書**（公開鍵と身元のバインド）で相手を検証する。サービス間の「この接続は誰か」を**ネットワーク層**で強く縛るときに使う。**TLS**（Transport Layer Security：HTTPSの暗号・身分確認の土台）の拡張。

- **実際のプロダクトでの役割**  
  **ゼロトラスト**（ネットワーク内でも常に検証する考え方）、**サービスメッシュ**（サイドカーでmTLSや観測を統一）、社内APIの横断認証。運用コスト（発行・ローテ・**失効**（Revocation：証明書の無効化））が高い。

- **似た概念の違い**  
  **JWT** はアプリ層のクレーム。**mTLS** は**TLSハンドシェイク**（暗号スイート交換と身分確認の手順）での**双方の鍵**。**VPN**（閉域ネットワーク）は別の閉域モデル。

- **なぜ本当に必要か**  
  **「社内IPだから信頼」** はクラウドとリモートワークで崩れた。**接続そのものに強い身元**を結びつけ、**ネットワーク境界が薄い**環境でも **サービスID** を検証するため。

- **ないと何が困るか**  
  **平文またはサーバ片側TLSだけ**の内部通信は、横移動で **なりすまし** されやすい。**VPN** 依存だと **ルーティングと運用** が重く、マイクロサービス単位のポリシーと噛み合わない。

- **歴史・背景と広まり方**  
  **公開鍵基盤（PKI）** と **TLS**（**RFC 5246** 以降の TLS 1.2 普及期）で「サーバ証明書」が標準化されたのに対し、**クライアント証明書**は企業 **イントラ** で長く使われた。**2010年代後半〜** **ゼロトラスト** と **サービスメッシュ**（**Istio** 等が **SPIFFE** 身分とセット）で「サービス間も証明書で相互認証」がクラウドネイティブの選択肢に載った。

- **プロの扱い方**  
  - 短い寿命 + 自動ローテ（**SPIFFE**/**SPIRE**（ワークロード身元の標準と実装）等の検討）  
  - 失効（**CRL**（失効リスト）/**OCSP**（オンライン照会））と配布経路のセキュリティ  
  - L7の**認可**とは役割分担（「入れた者」の中で何を許すかは別レイヤ）  

- **プロになるための質問**  
  - mTLS まで張ったあと、**まだ**アプリ層でJWTや**RBAC**（Role-Based Access Control：ロール基づく権限）が必要な理由は何か。  
  - 証明書ローテ時に「古いクライアントだけが残る」**グレース期間**（猶予）をどう設計するか。  
  - **サービスメッシュ**外のレガシクライアントはどう段階的に移行するか。  

---

## データ取得・キャッシュ・DB

### Pagination

- **現場のプロの解説**  
  大量リストを**分割取得**する。`OFFSET`/`LIMIT`（SQLの行スキップと件数制限）は深いページでDBが辛くなりやすく、**キーセット**（**カーソル**：前ページ最後の値を次の条件にする方式）が推奨される場面が多い。

- **実際のプロダクトでの役割**  
  APIの安定した応答時間、モバイルの**無限スクロール**（末尾で追加読み）。並行更新があると**抜け・重複**が起きうるためカーソル設計が重要。

- **似た概念の違い**  
  **キャッシュ**や**検索**のページング問題（一貫した**スナップショット**（ある時点の切り取り））とは別論点だが、UI/API契約としてはセットで決める。

- **なぜ本当に必要か**  
  **全件を一度に返す**と **メモリ・転送時間・タイムアウト** が一発で破綻する。**ユーザー体験（無限スクロール）** と **サーバ保護** の両方に必要。

- **ないと何が困るか**  
  **API・DB・モバイルアプリの同時クラッシュ**（巨大レスポンス）、**運用で LIMIT を手で切る**だけだと **深いページや並行更新** で不整合。

- **歴史・背景と広まり方**  
  **SQL の `OFFSET`/`LIMIT`**（**ISO SQL** に近い形で各DBが実装）が一覧の定番だったが、**深いページの非効率**が問題視され、**Twitter 等のタイムライン**を契機に **カーソル（since_id 等）** が「モバイル・無限スクロール」の標準になった。**GraphQL の connections（cursor）** も同系統の抽象化。

- **プロの扱い方**  
  - ソートキーの一意性（**タイブレーク列**（同値時の第2ソートキー））  
  - `OFFSET` 地獄（深いページでフルスキャンに近くなる）を**プロファイル**（性能測定）で確認する  
  - **総件数**取得のコストを別扱いにするか方針化  

- **プロになるための質問**  
  - 一覧の途中で行が挿入・削除されたとき、カーソル方式で**重複・欠落**をどう説明するか。  
  - 「次ページ」だけでなく「先頭へ戻る」要件はあるか、キャッシュと整合するか。  
  - 大量**エクスポート**（一括出力）と一覧APIを同じクエリに寄せていいか。

### Cache stampede（キャッシュスタンピード）

- **現場のプロの解説**  
  **キャッシュミス**（キャッシュに無く起源へ取りに行くこと）が**同時多発**し、全リクエストがDBへ落ちて雪崩になる現象。**ホットキー**（アクセス集中キー）の**TTL**（Time To Live：キャッシュの寿命）切れで典型。

- **実際のプロダクトでの役割**  
  読み取り多い設定・特売ページ・**バイラル**（拡散）コンテンツで顕在化。**可用性**とDBの問題に直結。

- **似た概念の違い**  
  **Cache invalidation** は鮮度の戦略。**スタンピード**は**同時再計算**の事故。対策は別（**ロック**／**確率的早期再計算**／**シングルフライト**（同時1本化））。

- **なぜ本当に必要か（対策として）**  
  キャッシュは **読みを速くする**代わりに **TTL切れや同時ミス** で **原点（DB）へ一斉落下**しうる。**その雪崩だけ**を防ぐパターン・実装が要る。

- **ないと何が困るか**  
  **一瞬でDBが過負荷**、**レイテンシが飛ぶ**、**キャッシュがあるのに全体が落ちる**という逆説的障害。ホットキーほど顕著。

- **歴史・背景と広まり方**  
  **CPU キャッシュ**の **miss storm** に似た現象として **Web キャッシュ**の文脈で **stampede** と呼ばれるようになった（**Zygmunt 等の研究**や大規模サイトのポストモーテムで有名）。**CDN・Redis** のホットキーと **TTL** の組み合わせで実務上の典型となり、**singleflight**・**probabilistic early expiration** などの対策語が定着した。

- **プロの扱い方**  
  - **singleflight**（Go等の同一キー再計算の合流） / **mutex**（相互排他ロック）による1本化  
  - **ガウシアンJIT**（期限前にランダムで再計算）やTTLに**ジッタ**（ランダム揺らぎ）  
  - **プリウォーム**（事前温め）、**永続キャッシュ**階層  

- **プロになるための質問**  
  - TTL切れの瞬間に**QPS**（Queries Per Second：秒あたりクエリ数）が何倍に跳ねるか、負荷試験で見積もったか。  
  - singleflight が失敗したとき、待機リクエスト全員に同じエラーを返すのは許容か。  
  - ホットキーを検知したら、TTL・キー分割・**プロテクション**（保護的再計算）のどれを先に手を打つか。  

**詳細（キャッシュ全般）:** [Caching Strategies](./17_common_patterns/caching_strategies.md)

### Cache invalidation

- **現場のプロの解説**  
  「キャッシュは**いつ嘘を許すか**」の設計。**キャッシュ無効化**（**cache invalidation**：いつ・どのキーを捨てるか）は難しいとされるが、実務では **TTL + イベント駆動無効化 + バージョンキー** で妥協点を探す。

- **実際のプロダクトでの役割**  
  カタログ、権限、設定の反映遅延とUXのトレードオフ。**強整合**（常に最新に近い）が要る読み取りはキャッシュを諦めるか、短TTLにする。

- **似た概念の違い**  
  **Cache stampede** はタイミングの同時性。**無効化**は**いつキャッシュを捨てるか**のルール。

- **なぜ本当に必要か**  
  キャッシュを使う以上、**DBの真実とコピーのズレ**は必ず生じる。**いつ・どの範囲で**古くてよいかを決めないと、**UX・在庫・権限**が壊れる。

- **ないと何が困るか**  
  **古い価格・権限・在庫**を表示し続ける、または **毎回DB読み**でキャッシュの意味がない。**無効化イベント欠落**で永遠に古いまま、など。

- **歴史・背景と広まり方**  
  **マルチレイヤキャッシュ**（CPU→メモリ→…）の古典問題として知られ、**Phil Karlton の「キャッシュ無効化は難しい」** ジョークで業界に刻まれた。**CDN の purge**、**HTTP Cache-Control / ETag**、**イベントソーシング**と組み合わせた無効化が、Web API 時代の設計論になった。

- **プロの扱い方**  
  - キー設計を**エンティティ**（ドメインオブジェクト単位）境界に揃える  
  - 書き込み後の無効化メッセージの**重複・順不同**に耐える  
  - 「**最終的に一貫**」（**eventual consistency**：収束するまで古い値がありうる）か「読み取りごと強い」かを要件化する  

- **プロになるための質問**  
  - 書き込み成功後に無効化イベントが失われたら、システムはどう復旧するか。  
  - キャッシュとDBの**読み取り一貫性**をユーザーにどう説明するか（「少し古い」許容）。  
  - バージョン番号（**ETag**（実体タグ：HTTPキャッシュ検証））とTTL-only、どちらを選ぶ基準は。  

**詳細:** [Caching Strategies](./17_common_patterns/caching_strategies.md)

### Composite index（複合インデックス）

- **現場のプロの解説**  
  複数列の**順序付き**インデックス（B-Tree等で先頭列から比較）。`(tenant_id, created_at)` のように、**左端列からの絞り込み**に効く。順序を間違えると効かない。

- **実際のプロダクトでの役割**  
  **マルチテナント**（複数顧客を同一DBで区画分け）、時系列、ステータス別一覧のクエリ性能を決める。**カーディナリティ**（列の値の種類の多さ）と**選択度**（条件で絞れる割合）の見立てが鍵。

- **似た概念の違い**  
  **Query optimization** は全体（**統計情報**・**JOIN**（結合）・**実行プラン**）。**複合インデックス**はその中核パーツ。**Sharding** はインデックス以前にデータ配置の問題。

- **なぜ本当に必要か**  
  よく使う **複合条件**（テナント＋日付 等）に **索引が乗らない**と、**フルスキャン**になりデータ量に直線的に劣化する。**検索の入口の幅**を絞るため。

- **ないと何が困るか**  
  一覧・ダッシュボードが **数秒〜タイムアウト**、**ディスクI/OとCPUが常時赤**、**インデックスなしのALTER** で一気にサービス影響。

- **歴史・背景と広まり方**  
  **関係モデル**と **B木／B+木インデックス**（**1970年代〜**）とともに登場。**複合キー**は「先頭列からの検索」という **B+木の性質**に合わせた設計論として定着。**カバリングインデックス**は **SQL Server の INCLUDE** 等で製品化され、**オプティマイザ**と **統計情報** の進化とセットで扱われてきた。

- **プロの扱い方**  
  - `EXPLAIN` / `EXPLAIN ANALYZE`（実測付き実行計画）で実測  
  - 書き込み増・ストレージ増とのトレードオフ  
  - **カバリングインデックス**（**INCLUDE**（PostgreSQL等）：非キー列を葉に持つ）の検討  

- **プロになるための質問**  
  - 複合インデックスの列順を `(A,B)` と `(B,A)` で変えると、どのクエリが効かなくなるか。  
  - **部分インデックス**（WHERE句付きの部分範囲インデックス）を使う条件は何か。  
  - インデックスが増えすぎたとき、どの指標で「削除候補」を決めるか。

### Query optimization

- **現場のプロの解説**  
  **スロークエリ**（遅いSQL）は**インデックス**だけでなく統計情報、N+1、不要**ソート**、**ロック待ち**、ネットワーク、**接続プール**（DB接続の再利用キュー）待ちが混ざる。**計測なしの最適化**を禁ず。

- **実際のプロダクトでの役割**  
  **p95**/**p99**（パーセンタイル：**レイテンシ**（応答時間）分布の上位側）改善、コスト、**レプリカ遅延**（読み取り複製の追いつき）の抑制。**ORM**（Object-Relational Mapping）の**生SQL**（手書きSQLの逃げ道）エスケープハッチと共存させる。

- **似た概念の違い**  
  **複合インデックス**は手段の一つ。**Sharding** は単体DBの限界を超える話。

- **なぜ本当に必要か**  
  遅さの原因は **SQL一つとは限らない**（統計・ロック・プール・N+1）。**計測でボトルネックを特定**し、**コストとUX** に直結する部分に効く投資をするため。

- **ないと何が困るか**  
  **p99 が悪化しても気づかない**、**RDSコストだけが増える**、**「インデックス足した」だけで改善せず**原因が不明のまま。

- **歴史・背景と広まり方**  
  **System R** 等の **クエリオプティマイザ** 研究（**1970年代〜**）から、**コストベース最適化** が RDB の中核に。**スロークエリ**対策は **Oracle・PostgreSQL・MySQL** の **EXPLAIN** 文化として普及し、**APM**（**New Relic** 等）で本番 **トレース** と結びついた。

- **プロの扱い方**  
  - スロークエリログ、**実行計画**の定期レビュー  
  - **バッチサイズ**（一度に取る行数）、JOIN順、部分インデックス  
  - 読み取りと書き込みの競合（**ロック**：排他制御）を見に行く  

- **プロになるための質問**  
  - p99 が悪いのは「そのクエリ」か「**プール待ち**」か、切り分け手順は。  
  - ORM が発行するSQLを、どのタイミングで EXPLAIN するか（本番相当データで）。  
  - **統計情報**（オプティマイザがプランを選ぶ材料）の更新遅れが疑わしいとき、何を確認するか。

### Sharding（シャーディング）

- **現場のプロの解説**  
  データを**シャードキー**（分割の基準列）で**水平分割**（行を機械間で分ける）し、単一ノードの限界を超える。**再シャーディング**・**局所障害**（一部シャードのみ停止）・**分散トランザクション**（複数シャードにまたがるACID）のコストが付きまとう。

- **実際のプロダクトでの役割**  
  巨大テーブル、テナント隔離、地理分散。アプリ層**ルーティング**（どのシャードへ送るか）か、**Vitess**（MySQL向けシャーディング基盤）等のミドルウェアかを選ぶ。

- **似た概念の違い**  
  **レプリケーション**（Replication：同一データのコピー）はコピー、**シャーディング**は分割。**複合インデックス**は各シャード内の話。

- **なぜ本当に必要か**  
  **1台のディスク・メモリ・接続数** にデータと負荷が乗り切らないとき、**水平分割**しかない（垂直スケールにも限界）。**スケールアウトの単位**を作るため。

- **ないと何が困るか**  
  **バックアップ・メンテ・障害が全件に直撃**、**ホットスポット**で特定シャードだけ炎上、**クロスシャードJOIN** なしにデータモデルを組めない。

- **歴史・背景と広まり方**  
  **水平分割**は **Google Bigtable 論文（2006）** や **NoSQL ブーム**で一般化。**MySQL 手動シャード**から **Vitess**、**Citus**、**MongoDB sharded cluster** へと「運用を製品が吸収」する流れになった。**分散SQL**（**Spanner**、**CockroachDB** 等）は「シャードを隠蔽」する方向へ進化している。

- **プロの扱い方**  
  - シャードキー変更の難しさを最初に見る  
  - **クロスシャードJOIN**（複数シャードにまたがる結合）を避けるデータモデル  
  - **バックフィル**（過去データの埋め戻し）・**マイグレーション**（移行）の手順  

- **プロになるための質問**  
  - シャードキーに `user_id` だけを選ぶと、どのクエリが困るか。  
  - 1シャードが熱くなったとき（**ホットスポット**）、再分割の手順は。  
  - 分散トランザクションを避けるため、ドメインをどう**境界づけ**（**Bounded Context**）するか。  

**詳細:** [Database Sharding](./17_common_patterns/database_sharding.md)

## 分散理論・トランザクション

### CAP 定理（実務的な読み方）

- **現場のプロの解説**  
  分散システムでは**一貫性**（Consistency：複製間の整合の厳しさ）・**可用性**（Availability：リクエストに応答し続けること）・**分断耐性**（Partition tolerance：ネットワーク分断があっても前進すること）を同時に最大限満たせない、という**思考の枠組み**。現場では「CP/APの二択」より、**分断時の振る舞い**（可用性を落として整合を取るか）を具体化する方が実用的。

- **実際のプロダクトでの役割**  
  **リージョン**（地理的な配置単位）間**レプリケーション**、**クォラム**（Quorum：過半数合意）・**リーダー選出**（どのレプリカが書き取り担当か）、**読み取り一貫性レベル**（**線形化可能性**（Linearizability）等）の説明責任に効く。

- **似た概念の違い**  
  **ACID** は主に単一DBトランザクション。**CAP** は複製と**ネットワーク分断**（Partition：ノード間通信の切断）の話。

- **なぜ本当に必要か**  
  **分散では「常に最新かつ常に応答」** を機械的に両立できない場面がある。**何を捨てるか**をチームで言語化しないと、**分断時の挙動**が事故と見なされる。

- **ないと何が困るか**  
  **ネットワーク障害のたびに「バグ」扱い**、**読み取りが古いのは許容か**が合意なく、**SLOと設計が不一致**。

- **歴史・背景と広まり方**  
  **Eric Brewer が2000年の PODC で講演**（**2002年頃にCAPと呼ばれる**）し、分散データストアの設計語彙になった。**2012年に Gilbert & Lynch が「分断時の二者択一」として整理**され、**NoSQL vs RDB** 論争の参照枠になった。現場では **「3つから2つ」より分断時の振る舞い** と **PACELC** で語る方が実用的、という反省も共有されている。

- **プロの扱い方**  
  - 「最終的に一貫」で十分なドメインかを切り分ける  
  - 分断時にエラーを返すか古い値を返すかをSLOと合意する  
  - **PACELC**（分断時/平常時の拡張モデル）等、拡張モデルで議論を精密化する  

- **プロになるための質問**  
  - ネットワーク分断が起きたとき、読み取りは「古い**レプリカ**」から返すか「エラー」か、どちらを選ぶか。  
  - 「CPシステム」と言い切るとき、具体的にどの一貫性モデル（線形化可能性など）を指すか。  
  - CAPを「3つから2つ選ぶ」だけで議論が足りない理由は何か。  

---

### ACID

- **現場のプロの解説**  
  **原子性**（Atomicity：全部成功か全部失敗か）・**一貫性**（Consistency：制約を満たす）・**分離性**（Isolation：同時実行の見え方）・**永続性**（Durability：コミット後は消えない）。単一の**トランザクションエンジン**内では強力だが、**分散トランザクション**ではコストと可用性のトレードオフが跳ねる。

- **実際のプロダクトでの役割**  
  金融・在庫などの強い整合が要る領域では **RDB**（Relational DB：関係DB）のトランザクションが第一選択になりやすい。マイクロサービス横断は **Saga**（長い業務を補償トランザクションで繋ぐ） / **段階的整合** と併記される。

- **似た概念の違い**  
  **CAP** は分散レプリカの話。**ACID** は（主に）単一DBの境界。**BASE**（Basically Available, Soft state, Eventual consistency：可用性と最終一貫性寄り）とは対照的に語られる。

- **なぜ本当に必要か**  
  **複数ステップを「全部成功か全部失敗」** に束ね、**中間状態をユーザーに見せない**ため。金融・在庫など **不変条件** が金銭や安全に直結する領域では不可欠。

- **ないと何が困るか**  
  **二重計上・残高不整合・在庫マイナス**、**部分だけ成功したログ**が残り **手戻り・手作業** が常態化。

- **歴史・背景と広まり方**  
  **Jim Gray** らによる **トランザクション処理**の確立（**1970〜80年代**）で **ACID** という語が定着（**Haerder & Reuter 1983** などで整理）。**RDB** が金融・在庫の基盤となった。**分散トランザクション（2PC）** は **XA** や **Google Spanner** で進化する一方、**マイクロサービス**では **Saga** が現実解として広まった。

- **プロの扱い方**  
  - **分離レベル**（**RC**（Read Committed）/**RR**（Repeatable Read）/**Serializable**（直列化可能））の要件とデッドロック  
  - 長いトランザクションの回避  
  - 分散協調が必要なら**2PC**（Two-Phase Commit：二相コミット）の代償を理解する  

- **プロになるための質問**  
  - 「一貫性（C）」がACIDとCAPで指すものが違う、と説明できるか。  
  - Serializable にするとデッドロックが増える可能性があるが、それでも選ぶケースは。  
  - マイクロサービス横断でACIDを捨てるとき、**どの不変条件**（**インバリアント**：常に真であるべき条件）を**サガ**で補うか。

## 信頼性パターン

### Circuit breaker（サーキットブレーカー）

- **現場のプロの解説**  
  下流が不健康なときに**呼び出しを早めに止め**、**スレッド枯渇**（スレッドプール占有）や**タイムアウト連鎖**（待ちが連鎖的に増える）を防ぐ。**半開**（Half-Open：試しに一部だけ通す状態）で回復を試す。

- **実際のプロダクトでの役割**  
  外部API、DB、他サービス呼び出しの前に置く。**フォールバック**（代替応答：キャッシュ値・**劣化モード**（機能を落とす））とセット。

- **似た概念の違い**  
  **Retry**（再試行）は再試行。**CB** は「試すのをやめる」。**Backpressure** は流量、**CB** は失敗状態への反応。

- **なぜ本当に必要か**  
  下流が遅い・壊れているときに **リトライだけ** すると **スレッドと接続が枯れ**、**障害が連鎖**する。**早めに遮断**し **資源を守る**ため。

- **ないと何が困るか**  
  **カスケード障害**（一つが全サービスを遅延）、**タイムアウトの連鎖**、**復旧してもまだ詰まっている**。

- **歴史・背景と広まり方**  
  電気の **サーキットブレーカー**（過電流で遮断）の比喩。**Michael T. Nygard『Release It!』（2007）** でソフトウェアパターンとして紹介され、**Netflix Hystrix**（**2010年代**）が JVM 界隈に普及。**Envoy**、**Polly**、**Resilience4j** へと言語横断で標準化した。

- **プロの扱い方**  
  - 失敗閾値・時間窓・半開間隔をメトリクスで調整  
  - タイムアウトより手前で失敗とみなす基準  
  - **べき等**でない処理との組み合わせに注意  

- **プロになるための質問**  
  - サーキットが**開**（Open：遮断）いたあと、最初の試行リクエストは誰が負担し、失敗したらどうなるか。  
  - リトライとサーキットを併用するとき、**指数バックオフ**（再試行間隔を指数的に延ばす）が悪化を招くことはあるか。  
  - フォールバックが古いキャッシュを返すとき、ユーザーへの表示はどう誤魔化さないか。  

**詳細:** [Circuit Breaker](./17_common_patterns/circuit_breaker.md)

### Idempotency（べき等性）

- **現場のプロの解説**  
  **同じ操作を複数回適用しても、結果が増えない**こと。ネットワークの**再送**、Webhook、決済、**メッセージキュー**と不可分。

- **実際のプロダクトでの役割**  
  `Idempotency-Key`（HTTPヘッダで重複を識別）、**ユニーク制約**（DBの一意制約）、**状態機械**（State Machine：状態と遷移のモデル）での重複イベント吸収。ユーザー体験（二重課金防止）に直結。

- **似た概念の違い**  
  **Circuit breaker** は障害の伝播制御。**べき等性**は**副作用**（課金・在庫減など）**の重複**への耐性。**トランザクション**とは別（後者は**原子性**）。

- **なぜ本当に必要か**  
  **ネットワークは再送する**、**ユーザーは二重タップする**、**Webhookは再送する**。副作用のある操作を **何度やっても壊れない**ようにするため。

- **ないと何が困るか**  
  **二重課金・二重在庫減・二重登録**、**顧客対応と返金コスト**、**データ修正バッチ**が常態化。

- **歴史・背景と広まり方**  
  数学の **べき等**（**f(f(x))=f(x)** など）から来る概念が、**分散システム**では **「少なく一度」** セマンティクスと結びついた。**HTTP の GET/PUT の意味論**、**決済の二重投稿**、**Stripe の Idempotency-Key** などが実務で定着し、**メッセージの Exactly-once 幻想**に対する現実解として語られる。

- **プロの扱い方**  
  - キーのスコープ（ユーザー単位かグローバルか）  
  - 保存期間と衝突時のレスポンス（同じ結果を返す）  
  - GETの**安全性**（何度呼んでも壊さない）・POSTの**非べき等**をAPI設計で区別する  

- **プロになるための質問**  
  - 「べき等」と「同じレスポンスを返す」は常に同じか、違う例は。  
  - 決済のような副作用で、**状態機械のどの遷移**がべき等キーで守られるか。  
  - クライアントがべき等キーを生成する際、**衝突確率**（UUID等）をどう見積もるか。  

**詳細:** [Idempotency](./17_common_patterns/idempotency.md)

## 並行性・低レベル・セキュリティ

### Livelock（ライブロック）

- **現場のプロの解説**  
  **デッドロック**と違い、**スレッド**は動いているが**実質的に前に進まない**状態。互いに譲り合って再試行を繰り返すパターンで起きやすい。

- **実際のプロダクトでの役割**  
  **ロックフリー**（ロックなし並行）アルゴリズムの誤用、過度な再試行、**リトライ嵐**。観測上はCPUを食うがスループットが出ない。

- **似た概念の違い**  
  **Deadlock**（デッドロック：待ちの循環）は待ちの輪で停止。**Starvation**（飢餓：順番が回ってこない）は特定タスクが飢える。**Livelock** は「動いてるが無意味」。

- **なぜ本当に必要か（概念として）**  
  **再試行・譲り合い**だけでは進まない並行バグを**区別**する語彙。**デッドロック検知が無反応**なのにスループットゼロ、を説明する。

- **ないと何が困るか**  
  **CPUを食うのに処理件数が増えない**、原因を **デッドロック**と誤診して **間違った対策**。

- **歴史・背景と広まり方**  
  **並行処理・オペレーティングシステム**の教科書的トピック（**Dijkstra の銀行家アルゴリズム**等と同列）。**分散ロック**や **楽観的並行性制御**の **リトライ** が絡むと実務で観測され、**「デッドロックではないのに進まない」** として区別されるようになった。

- **プロの扱い方**  
  - ランダムバックオフ（**ジッタ**：ランダムな待ち）  
  - 再試行上限と全体タイムアウト  
  - 競合メトリクス（失敗率・再試行率）の監視  

- **プロになるための質問**  
  - デッドロック検知ツールが「待ちなし」と出るのにスループットが出ないとき、何を疑うか。  
  - 協調的な再試行が「互いに道を譲り合う」コードに見えるとき、どうログで証明するか。  
  - ライブロックと**スレッド飢餓**（**Starvation**）の違いを、運用メトリクスで説明できるか。

### False sharing（偽共有）

- **現場のプロの解説**  
  別変数でも**同一キャッシュライン**（CPUがメモリを読み込む最小単位の塊、例64バイト）に乗ると、**CPUコア**間でラインが無駄に**無効化**（他コアの書き込みでキャッシュ一貫性のため無効）され**性能が崩れる**。他の分散用語とはレイヤが異なる（**マルチスレッド**・低レベル最適化）。

- **実際のプロダクトでの役割**  
  高並列カウンタ、**パディング**（余白でラインを分離）、**アライメント**（アドレス境界揃え）、**パーコア集計**（コアごとに集計してから合算）で回避。数値計算・ゲームエンジン・DBエンジン開発で実務。

- **似た概念の違い**  
  **キャッシュスタンピード**はアプリのキャッシュ層（Redis等）。**False sharing** は**CPUキャッシュ**ライン。混同しない。

- **なぜ本当に必要か（概念として）**  
  **並列化したのに遅い**とき、**アルゴリズムは正しい**のに **メモリ上の配置**で性能が落ちることを説明するため。

- **ないと何が困るか**  
  **スケールアウトしても遅い**、**プロファイルしないと一生見つからない**ボトルネック。

- **歴史・背景と広まり方**  
  **マルチプロセッサ**普及（**1990年代〜**）に **MESI 等のキャッシュ一貫性プロトコル** が一般化し、**別変数でも同一キャッシュラインなら競合**する現象が問題になった。**Java `@Contended`**、**C++ アライメント**、**並列数値計算**の最適化で知られる。

- **プロの扱い方**  
  - **プロファイラ**（性能プロファイルツール）で**キャッシュミス**を確認  
  - ホットなカウンタの分割（**シャーディング**語と同名だがCPU話）  
  - 早すぎる最適化を避け、計測後に手を入れる  

- **プロになるための質問**  
  - **キャッシュライン**サイズ（例: 64B）を意識したとき、配列のどの配置が危険か。  
  - false sharing を疑うとき、まずどのプロファイル指標（**CPU cycle**、**cache miss**）を見るか。  
  - アプリの「キャッシュスタンピード」と、CPUの「キャッシュライン競合」を混同しないための言い方は。

### CSRF（クロスサイトリクエストフォージェリ）

- **現場のプロの解説**  
  ブラウザが保持する**セッション**や**Cookie**（HTTP Cookie：保存された小さな名値）を悪用し、受害者の意図しない**状態変更**リクエストを送らせる攻撃。

- **実際のプロダクトでの役割**  
  CookieベースセッションのフォームPOST、設定変更系エンドポイントが標的。**SameSite**（Cookieを同一サイトに限定する属性）、**CSRFトークン**（フォームに隠しトークン）、**カスタムヘッダ**要求が対策の定番。

- **似た概念の違い**  
  **XSS**（Cross-Site Scripting：スクリプト注入）は別の脅威（だがCSRF対策を迂回することがある）。**CORS**（Cross-Origin Resource Sharing：ブラウザのクロスオリジン読み取り制御）は読み取り制限でありCSRFの銀弾ではない。

- **なぜ本当に必要か**  
  ブラウザは **同一サイトに見えるリクエスト**に **Cookieを勝手に付ける**。攻撃者サイトから **受害者の権限で**状態変更POSTが飛ぶのを防ぐため。

- **ないと何が困るか**  
  **勝手に送金・設定変更・退会**、**ユーザーは操作していない**のに被害。**ログ上は正規セッション**で追いにくい。

- **歴史・背景と広まり方**  
  **セッションCookie がブラウザに自動添付**される **Web** 特有の攻撃として **2000年代前半から OWASP** で整理された。**CSRF トークン**、**Referer チェック**、**SameSite Cookie**（**RFC 6265bis**、**Chrome のデフォルト化**）へと対策が進化。**REST API** では **Cookie ではなく Bearer** を使う設計と役割分担される。

- **プロの扱い方**  
  - 状態変更はPOST/PUT/DELETE、GETを**安全**（副作用なし）に  
  - Cookie の `SameSite`、`Secure`（HTTPS限定）、`HttpOnly`（JSから読めない）  
  - 重要操作は再認証・**WebAuthn**（FIDO2等の強い認証）等の追加因子  

- **プロになるための質問**  
  - 「Cookieが自動で付く」ことがCSRFの根拠だが、`Authorization: Bearer` だけのAPIは安全か、なぜか。  
  - **SameSite=Lax** のとき、どの**ナビゲーション**（トップレベル遷移等）でCookieが送られるか。  
  - CSRFトークンを二重送信する場合、トークンを盗む攻撃（XSS）との関係は。
