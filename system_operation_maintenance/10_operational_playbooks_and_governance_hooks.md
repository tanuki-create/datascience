# 10 — 運用 Playbook・RACI・ガバナンス接続

本章は、[04](./04_observability_sre_and_incidents.md)（障害・SRE）と [02](./02_delivery_and_contracts.md)（境界）を **そのまま使える形** に落とす **ひな形**です。数値（分・%）はすべて **貴社の SLA/SLO に差し替え**してください。

## 1. この章の使い方

## 2. 重大度（Severity）ひな形 — 一般＋AI 拡張

重大度は **ユーザー影響・データ・法令**の **最大**で決める。曖昧なら **一つ上げて**初動し、後で下げる。

| レベル | 名称（例） | ユーザー影響 | データ・法令 | 初動目標（例・社内SLO） |
|--------|------------|--------------|--------------|-------------------------|
| **P1** | 重大 | 本番の主要機能が **全ユーザー**で利用不可、または誤実行が進行中 | 大量漏えい疑い・越権の大規模疑い | 〇分以内にオンコール着手、〇時間以内に **緩和策** |
| **P2** | 高 | 主要機能が **著しく劣化**（多くのユーザ）、一部利用不可 | 限定的だが **機密区分**に関わる | 〇時間以内に緩和方針 |
| **P3** | 中 | 一部機能・一部ユーザ、回避策あり | 軽微または未確認 | 営業日内対応 |
| **P4** | 低 | 軽微、問い合わせ・改善 | 該当なし | バックログ |

**AI 拡張の「P1 候補」**（技術的には生きていても、事業・コンプラで P1 扱いにすることがある）

- **誤った実行**（メール一斉送信、誤承認、誤データ書込）が止まらない／巻き戻し不能。
- **越権応答**（本来見えない文書の引用・要約）が **再現する**。
- **プロンプトインジェクション**や **ツール濫用**が本番で再現し、危害が拡大しうる。
- **コスト暴走**が一日で予算を超過し、**止められるスイッチ**がない（財務・契約上の重大）。

**プロのコツ**: 「502 が出ない」障害ほど **重大度を上げ忘れる**。品質・安全は **別 SLI** で監視する（→ [04](./04_observability_sre_and_incidents.md)、[05](./05_ai_specific_ops.md)）。

## 3. インシデント初動 — 切り分け順（チェックリスト）

**最初の 15 分でやること**（並列可）

1. **影響範囲** — 全テナントか、特定部門か、特定機能か。
2. **同時イベント** — 直近デプロイ、インデックス更新、モデル/API 設定変更、IdP/ネットワークメンテ。
3. **相関ID** — 代表トレースを1本確保（以降の議論の基準）。
4. **緩和優先** — kill switch（LLM停止・ツールOFF・読取専用縮退）が定義済みなら **先に実行可否を判断**。

**次の切り分け**（AI 固有の枝）

| 症状の匂い | まず疑う層 | 確認する SLI / 証跡 |
|------------|------------|---------------------|
| 突然の大量 429 / タイムアウト | 依存API・自前レート制限 | プロバイダーステータス、自前クォータ、リトライ嵐 |
| 「正しそうだが古い／嘘」 | RAG 鮮度・ACL・検索 | ゼロヒット率、インデックス世代、ACL同期遅延 |
| ツール失敗率だけ上がる | 権限・連携先・スキーマ | ツール別エラー、IdPトークン期限 |
| 請求だけ跳ねる | ループ・長文化・キャッシュ無効 | トークン/リクエスト分布、上位ユーザ |
| セーフティ拒否が急増 | ポリシー変更・入力分布 | ポリシーバージョン、入力長 |

詳細シナリオ例は §5。

## 4. ベンダー × 顧客 — 典型的 RACI（凡例）

**R/A/C/I** = Responsible / Accountable / Consulted / Informed。案件で **1セルずつ**確定させる。

| 作業領域 | 顧客（情報システム・セキュリティ） | ベンダー（開発・運用） |
|----------|-------------------------------------|-------------------------|
| IdP・SSO・端末ポリシー | **A/R** | C |
| アプリ・オーケストレーションコード | C | **A/R** |
| LLM プロバイダ契約・キー（顧客締結の場合） | **A/R** | C |
| 本番 VPC 内のランタイム（顧客クラウドの場合） | **A**（インフラオーナー） | **R**（デプロイ運用） |
| マスタデータ・文書ストアの正 | **A/R** | I |
| RAG 取込パイプライン（カスタムの場合） | C | **A/R** |
| アプリ監視・オンコール（契約による） | I / R | **A/R** |
| インシデント顧客報告文の事実確認 | **C** | **R** |

**争点になりやすいところ**: 「ログを見せて」の **範囲**、**パッチ適用**、**データ削除**。契約に無いと火種（→ [02](./02_delivery_and_contracts.md)、[07](./07_professional_tensions.md)）。

## 5. シナリオ別「初手」例（プレースホルダ）

状況に応じて Runbook を **リンク差し替え**。ここでは **意思決定の順序**だけ固定。

### 5.1 外部 LLM API 全面遅延・429

1. ステータスページ確認 → 自前の **リトライ・バックオフ**が失控していないか。
2. **キュー滞留**を可視化し、必要なら **受付停止**または **非同期化**をビジネスと合意。
3. **縮退モデル**や **キャッシュ**への切替（事前に定義済みなら）。

### 5.2 「見えてはいけない文書」が引用されたとの報告

1. **P1 扱い**でチケット化（証跡保全）。該当セッションの **相関ID・時刻・ユーザID**。
2. 再現手順が取れる範囲で **再現**（`*_test` ユーザで ACL 境界を叩く）。
3. 原因が **検索ACL** / **アプリバグ** / **プロンプト** のどれかで **別 Runbook** に分岐（→ [09](./09_pro_field_notes_2026.md) LLM02/08）。
4. 顧客・法務への連絡テンプレは **事前承認済み**のものを使う。

### 5.3 夜間にコストが暴騰

1. **請求ダッシュボード**と **トークン上位**を突合。
2. **キー漏洩・バッチ暴走・ループ**のどれか。キーなら **即失効**。
3. 再発防止：上限・異常検知・kill switch の **有無**をポストモーテムに入れる。

### 5.4 デプロイなしで「答えが変わった」

1. **モデルピン**が外れていないか、プロバイダの **デフォルト更新**が無いか。
2. **ナレッジ世代**・**プロンプト**の意図しない変更が無いか（設定ドリフト）。
3. **回帰評価**を即再実行し、差分をリリースノート扱いにする。

## 6. 引き渡し直後 30 日チェック（運用立ち上げ）

プロジェクト完了の「**技術完了**」と「**運用完了**」を分ける。

- [ ] 監視：主要 SLI が **本番で見えている**（ダミーではない）。
- [ ] Runbook：**P1 を1つ**タイマーなしでドライランした（欠けを棚卸し）。
- [ ] 連絡網：オンコールが **実際に当番**で応答した。
- [ ] 変更：**緊急リリース**の承認経路が1本通った。
- [ ] データ：**バックアップ/リストア**の担当と演習記録がある。
- [ ] AI：**評価セット**のオーナーと週次レビュー枠がカレンダーに乗った。
- [ ] コスト：**日次アラート**と **月次レビュー**が誰の承認か決まっている。
- [ ] セキュリティ：**OWASP LLM Top 10**のギャップがチケット化されている（→ [09](./09_pro_field_notes_2026.md)）。

## 7. 定例ミーティング — 最小アジェンダ

### 週次（15〜30 分）：運用・信頼性

- インシデント・未解決チケット（P2以上）
- SLI の週次トレンド（可用性・遅延・エラー・AI 固有）
- 直近リリースと **予定変更**
- ベンダー/クラウドの **メンテ予定**

### 月次（45〜60 分）：プロダクト・コスト・リスク

- 請求と **単価ドライバ**（トークン、検索、格納）
- 評価セット／本番サンプルからの **品質所感**
- サブプロセッサ・**モデル版**の変更予告の棚卸し
- セキュリティ：**脆弱性**、権限レビュー差分、インシデント是正の進捗

### 四半期：ガバナンス・演習

- **脅威モデル**またはデータ分類の見直し
- DR / リストア / **縮退モード**の演習
- AIBOM / **構成一覧**の更新（→ [06](./06_change_release_and_data_governance.md)）

## 8. サポートチケット分類（運用品質を上げる）

**Type（性質）** と **Area（層）** を必須項目にすると、ベンダーと顧客で **週報が揃う**。

| Type | 例 |
|------|-----|
| incident | 本番障害・疑義 |
| problem | 再発傾向・根本原因未 |
| change | リリース依頼・設定変更 |
| request | 機能・改善 |
| question | 使い方 |

| Area（カスタムAI向け） | 例 |
|------------------------|-----|
| platform | 自社IaaS/PaaS |
| app | オーケストレーション・アプリ |
| llm_api | 外部LLM |
| rag | 取込・検索・インデックス |
| identity | SSO・権限 |
| data | マスタ・文書ソース |
| cost | 課金・クォータ |

## 9. ガバナンスフレームとの「接続」の仕方（位置づけのみ）

法域・業種で要求が異なるため、**認証取得の手引きではない**。社内の **リスク言語**を揃えるための **マッピング先**として使う。

| フレーム | 公式の出発点（2026 時点の一般的参照） | 本資料との対応（例） |
|----------|----------------------------------------|----------------------|
| **NIST AI RMF** | [NIST AI Risk Management Framework](https://www.nist.gov/itl/ai-risk-management-framework) | Map/Measure/Manage に **SLI・評価・インシデント**を載せる（[04](./04_observability_sre_and_incidents.md)、[05](./05_ai_specific_ops.md)） |
| **ISO/IEC 42001** | [ISO/IEC 42001 — AI management system](https://www.iso.org/standard/81230.html) | **方針・リスク・変更・監査**のループと [06](./06_change_release_and_data_governance.md)、§7 定例を接続 |
| **OWASP GenAI** | [LLM Top 10（2025）](https://genai.owasp.org/llm-top-10/) / [Agentic Top 10（2026）](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) | **技術コントロール**の網羅チェック（[09](./09_pro_field_notes_2026.md) 冒頭の 2026 追従リスト） |
| **NIST AI 600-1** | [Generative AI Profile（DOI）](https://doi.org/10.6028/NIST.AI.600-1) | 生成AIリスクを RMF 関数に載せる **記述テンプレ**（[09](./09_pro_field_notes_2026.md)） |
| **EU AI Act** | [Implementation timeline（欧州委員会）](https://ai-act-service-desk.ec.europa.eu/en/ai-act/eu-ai-act-implementation-timeline) | 法務・DPO と **義務の該当性**を突合（技術資料は解釈代替にならない） |

**プロの作法**: フレームに **全部載せよう**とせず、既存の **SLI・チケット・承認ログ**に **どの節が対応するか** だけ表にすると監査対応が楽。

## 10. 大企業導入で詰まりやすい「社内手続き」リスト

技術以前に遅延する要因。**キックオフで日程を仮押さえ**する。

- 情報システム：**本番環境・VPC 接続・FQDN・WAF**
- セキュリティ：**データ分類、ログ出口、ペネト許可**
- 調達：**クラウドマーケットプレイス、注文残、ライセンス形態**
- 法務：**DPA、学習利用、サブプロセッサ、越境**
