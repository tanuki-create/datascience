# 02 — 引き渡し・契約・サポート境界

導入 **前〜中** に決め遅れると、本番で **「それは契約外」** が連発します。カスタムAIでは「バグ」と「挙動の好み」「データの古さ」が混ざるため、**成果物の定義**と**責務分界**を書面で硬くします。

---

## 1. 成果物の階層（何を「納品」と呼ぶか）

現場では次の層に分けて一覧化します。

| 層 | 例 | 備考 |
|----|-----|------|
| **実行環境** | IaC、K8s manifest、環境別設定 | 顧客VPC内か、ベンダー管轄か |
| **アプリケーション** | API、バッチ、UI | バージョンとロールバック単位 |
| **AI資産** | プロンプトテンプレ、ツール定義、評価セット、RAGパイプライン | Git等で版管理の対象にする |
| **データ・ナレッジ** | インデックス、コネクタ、ACL | **鮮度と権限**は運用で動く |
| **運用設計** | Runbook、オンコール、監視定義、障害分類 | 「納品」に含めるか契約で明記 |
| **セキュリティ** | 脅威モデル概要、ログ方針、鍵管理 | 監査で必ず聞かれる |

**素人が誤解しやすい点**: ソースコードを渡せば終わり。  
**現場**: 運用に載らないコードは **負債**。**観測・手順・責任者**までセットが製品。

---

## 2. 受け入れ基準（受入れテスト）の現実味

### 2.1 機能受入れ

- ユーザーストーリー／API仕様に対する **自動テストの有無と範囲**。
- **非機能**（同時接続、ピーク時遅延、バックプレッシャ）をどこまで試すか。

### 2.2 AI受入れ（別枠で必須）

**デモの良さ**ではなく、次を満たすか。

- **回帰セット**: 代表クエリ＋期待される **制約**（必ず引用する、断定しない、など）。
- **禁止事項テスト**: PII、越権、プロンプトインジェクション対策の **確認レベル** を合意。
- **説明責任**: 根拠が期待どおり **トレース可能** か（RAGならどの文書IDか）。

受入れで **「曖昧なGO」** を出すと、本番の問い合わせが **仕様変更交渉** に化けます。

## 3. 保証（Warranty）と保守の切り分け

典型的な切り分け（案件ごとに調整）:

| 種別 | 内容 | 備考 |
|------|------|------|
| **欠陥バグ** | 仕様通りに動かない、クリティカル脆弱性 | 修正優先度高 |
| **環境起因** | 顧客側IdP/ネットワーク/クォータ | 切り分け手順が要る |
| **第三者** | LLM API、検索基盤の障害・仕様変更 | SLAは **元製品に依拠**＋逃げ道 |
| **改善要望** | 応答のトーン、ナレッジ網羅、業務ルール追加 | 保守・別契約に寄せる |
| **データ品質** | OCR誤り、古い規程、権限ミス | 多くは **顧客データ責任** |

**争点**: 「バグ」か「データ」か「モデルの限界」か。  
→ **ログと評価セット**で裁く前提を契約に書く（→ [05](./05_ai_specific_ops.md)）。

---

## 4. SLA の読み方（数字の裏）

契約書の **稼働率** を採用する前に確認すること:

1. **測定対象** — どのエンドポイント／どのリージョンか。
2. **除外** — 計画メンテ、顧客原因、不可抗力、ベンダー指定の依存先。
3. **計測方法** — 合成監視か実トラフィックか、ウィンドウは月次か四半期か。
4. **補償** — クレジットか、延長保守か。上限キャップはあるか。

**プロのメンタルモデル**: SLAは **統計とプロセスの契約**。**絶対に止まらない**のではなく、止まったときの扱いが書いてあるもの。

## 5. サポート境界とエスカレーション

文書化すべき最小セット:

- **重大度定義**（全停止 / 一部機能 / 軽微 / 問い合わせ）
- **初期応答時間・復旧目標**（社内SLO。SLAと混同しない）
- **連絡チャネル**（チケット必須化、電話はどの重大度からか）
- **必要情報**（相関ID、時刻、入力の **再現可能な切り抜き** ― プライバシー配慮付き）

AI向けの一文を足すとよい:

- **入力例の提供**がセキュリティ上難しい場合の代替（ログ参照権限、マスキング手順）。

---

## 6. IP・データ・学習利用（必須の論点）

| 論点 | 確認内容 |
|------|-----------|
| **顧客データの扱い** | モデル学習に使うか、ログ保管期間、削除権 |
| **派生物** | ファインチューン済み重み、顧客専用インデックスの帰属 |
| **サブプロセッサ** | どのクラウド／APIにデータが出るか |
| **越境** | リージョン、DR時の扱い |

「将来的にベンダーが横展開に使う」話は **最も揉めやすい**。契約段階で **opt-in/opt-out** を固定する。

---

## 7. 引き渡し（ハンドオーバー）に含めるべきオンボーディング

1. **アーキテクチャ概要**と **責務分界図**（→ [03](./03_architecture_for_operations.md)）
2. **Runbook**（起動停止、秘匿ローテーション、リストア手順のリンク）
3. **オンコール連絡網**とエスカレーション
4. **監視ダッシュボード**と **定例レビュー**（週次/月次）
5. **変更手順**（→ [06](./06_change_release_and_data_governance.md)）
6. **運用に割く人員・FTE の前提**（オーナー・データ・情シス・保守契約の人日）— 目安は → [11](./11_staffing_and_capacity.md)

## 8. プロ同士で早期に対立しやすいポイント（先回り）

- **スコープクリープ**を「保守」か「追加開発」か。
- **第三者APIの仕様変更**の追従は誰の工数か。
- **データが汚い**ことによる精度不足の扱い。
