# 08 — 成熟度ロードマップ（基本から一流まで）

同一の技術スタックでも、**運用の成熟度**で可用性・セキュリティ・改善速度が決まります。レベルは絶対評価ではなく、**次の投資**の優先順位付けに使ってください。

---

## レベル定義（目安）

### レベル 0 — 手作業・個人依存

**状態**

- デプロイが手元マシン依存。設定が口頭・チャット。
- ログはバラバラ。障害対応は **その場のヒーロー**。
- AIの品質は **デモと手触り**で判断。

**リスク**

- 離脱・休暇で止まる。コンプラ証跡に弱い。

**次の一歩**

- リポジトリと **最低限のCI**。本番秘密を **共有ストア**に。

---

### レベル 1 — 再現可能だが「本番は脆い」

**状態**

- パイプラインでデプロイ可能。staging はある。
- 監視は **アップダウン**中心。オンコールは名ばかり。
- モデル・プロンプトの版が **曖昧**。

**リスク**

- 変更で壊す。外部APIの揺れに弱い。

**次の一歩**

- **相関ID**と主要 **SLI**。プロンプト・モデルIDの **BOM**（→ [05](./05_ai_specific_ops.md)）。

---

### レベル 2 — 運用が回り始める

**状態**

- Runbook とオンコールローテーション。**障害分類**がある。
- ステージングが **本番に近い**。ロールバックを演習したことがある。
- 回帰評価セットが **リリースゲート**に入っている。

**リスク**

- データガバナンス・サブプロセッサ変更が **後追い**になりがち。

**次の一歩**

- **データ分類とログ方針**の文書化。RAG の **ACL同期テスト**。

---

### レベル 3 — 計測駆動・変更が速い

**状態**

- SLO とエラーバジェット。ダッシュボードが **事業にも見える**。
- カナリア・フィーチャーフラグ。インデックス二段切替など **安全なデータ変更**。
- ポストモーテムが **チケットに繋がる**。game day を年次で実施。

**リスク**

- 組織が大きいと **合意形成コスト**がボトルネック（→ [07](./07_professional_tensions.md)）。

**次の一歩**

- **品質・コスト・安全**の統合レビュー（月次）。自動カナリ評価。

---

### レベル 4 — 一流（継続的改善が仕組み）

**状態**

- **実験**と**本番**の境界が明確。実験結果が **本番設定**に安全に昇格。
- トレースとログから **離脱理由の自動クラスタ**（可能な範囲で）。
- サプライチェーン・**脅威モデル**の定期更新。DR が **実測**されている。

**備考**

- 「一流」は **道具の購入**ではなく **意思決定と訓練の積み重ね**。規模が小さくてもレベル3〜4の **部分集合**は狙える。

---

## チェックリスト（自己評価用）

採点より **未チェックの重み**を見る使い方を推奨。

### 引き渡し・契約

- [ ] サポート境界・重大度・エスカレーションが文書化されている（→ [02](./02_delivery_and_contracts.md)）
- [ ] 第三者依存（LLM等）の **例外条項**が合意されている

### アーキテクチャ

- [ ] 環境分離と秘密ローテーション手順がある（→ [03](./03_architecture_for_operations.md)）
- [ ] ブラスト半径が意識された境界設計（テナント/VPC/キー）

### 観測・障害

- [ ] SLI/SLO が定義され、ダッシュボードがある（→ [04](./04_observability_sre_and_incidents.md)）
- [ ] Runbook とオンコールが実運用されている
- [ ] 年次（またはそれ以下）で復旧訓練をしている

### AI 固有

- [ ] モデル・プロンプト・インデックス世代のトレーサビリティ（→ [05](./05_ai_specific_ops.md)）
- [ ] 回帰評価とカナリ/段階リリース
- [ ] コスト急増アラートと上限設計

### データ・変更

- [ ] データ分類・保持・削除の方針（→ [06](./06_change_release_and_data_governance.md)）
- [ ] 変更管理に「非コード変更」（プロンプト・索引）が含まれる

### 組織

- [ ] 争点がテーブル化され、意思決定ログが残る（→ [07](./07_professional_tensions.md)）

### 2025〜2026 の強化項（任意だが差がつく）

- [ ] OWASP LLM Top 10（2025 版）を **リリースゲート**に紐づけている（→ [09](./09_pro_field_notes_2026.md)）
- [ ] エージェント／MCP 構成なら [OWASP Top 10 for Agentic Applications（2026）](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) と [Secure MCP Server Development](https://genai.owasp.org/resource/a-practical-guide-for-secure-mcp-server-development/) を **設計・受入れ**に組み込んでいる
- [ ] エージェント／ツールの **権限・承認・冪等性** が Runbook と監視に載っている

---

## 投資の優先順位（経験則）

1. **観測とロールバック**（壊しても戻せる）
2. **秘密とデータ出口**（一度の事故で終わらない）
3. **評価ゲート**（改善の速度を上げるには先に安全装置）
4. **自動化と実験文化**（成熟の後半で効いてくる）
