# 02. ユビキタス言語 - ドメインエキスパートと開発者の共通言語

## この章の学習目標

この章を読むことで、以下を理解できるようになります：

- ユビキタス言語とは何か、なぜ重要か
- ソーシャルメディアでのユビキタス言語の例

- 用語集の作成と維持方法

## ユビキタス言語とは

**ユビキタス言語（Ubiquitous Language）**は、DDDの最も重要な概念の一つです。

### 定義

> **ユビキタス言語**は、ドメインエキスパートと開発者が**共有する共通言語**です。コード、ドキュメント、会議、すべての場面で同じ用語を使用します。

### なぜ「ユビキタス（どこにでもある）」なのか

「ユビキタス」という言葉は、「どこにでも存在する」という意味です。ユビキタス言語は、以下のすべての場所で使用されます：

- **コード**: クラス名、メソッド名、変数名
- **ドキュメント**: 設計書、仕様書、APIドキュメント
- **会議**: ドメインエキスパートとの打ち合わせ
- **テスト**: テストコード、テストケース名

## ユビキタス言語がない場合の問題

### 問題1: 用語の不統一

```python
# 開発者Aのコード
class Message:
    def publish(self):
        # "message"という用語を使用
        pass

# 開発者Bのコード
class Post:
    def create(self):
        # "post"という用語を使用（同じものを指している）
        pass

# 開発者Cのコード
class Article:
    def submit(self):
        # "article"という用語を使用（これも同じもの）
        pass
```

**問題点**:
- 同じ概念に対して異なる用語が使われる
- コードレビューで混乱が生じる
- 新メンバーが理解しにくい

### 問題2: ドメインエキスパートとのコミュニケーションギャップ

**会議での会話例**:

```
ドメインエキスパート: 「ユーザーが投稿を作成したら、フォロワーに通知を送る必要がありますね」

開発者A: 「了解です。Messageエンティティに通知フラグを追加します」

ドメインエキスパート: 「Message？投稿のことを言っているのですが...」

開発者B: 「ああ、Postのことですね。Postエンティティに通知フラグを追加します」

ドメインエキスパート: 「Post？私たちは『投稿』と呼んでいますが...」
```

**問題点**:
- ドメインエキスパートと開発者の間で用語が一致しない
- 会議の時間が無駄になる
- 誤解が生じ、実装が間違った方向に進む

### 問題3: コードとドキュメントの不一致

```python
# コード
class UserMessage:
    def send(self):
        pass

# ドキュメント
# 「ユーザーが投稿を作成すると...」
# コードでは "UserMessage" だが、ドキュメントでは "投稿"
```

**問題点**:
- コードとドキュメントで用語が異なる
- 新メンバーが混乱する
- 保守性が低下する

## ユビキタス言語の確立

### ステップ1: ドメインエキスパートとの対話

ユビキタス言語は、ドメインエキスパートと開発者が**協働して**作り上げます。

#### 対話の例

```
開発者: 「ユーザーが作成するコンテンツのことを、何と呼びますか？」

ドメインエキスパート: 「『投稿』と呼んでいます。ユーザーが何かを投稿する、という意味です」

開発者: 「では、コードでも 'Post'（投稿）という用語を使います。他の用語はありますか？」

ドメインエキスパート: 「『メッセージ』という用語もありますが、それは別の意味です。メッセージは、ユーザー間で直接やり取りするものです」

開発者: 「了解です。'Post'（投稿）と'Message'（メッセージ）は別物として扱います」
```

#### 対話のポイント

1. **ドメインエキスパートの用語を尊重する**: 開発者が勝手に用語を決めない
2. **曖昧さを明確にする**: 「投稿」と「メッセージ」の違いを明確にする
3. **用語を記録する**: 対話の内容を用語集に記録する

### ステップ2: 用語集の作成

用語集（Glossary）を作成し、用語の定義を明確にします。

#### ソーシャルメディアの用語集例

| 用語 | 英語 | 定義 | 例 |
|------|------|------|-----|
| 投稿 | Post | ユーザーが作成する公開コンテンツ | 「今日はいい天気です」という投稿 |
| メッセージ | Message | ユーザー間で直接やり取りする非公開コンテンツ | ユーザーAからユーザーBへのメッセージ |
| フォロー | Follow | ユーザーが他のユーザーをフォローすること | ユーザーAがユーザーBをフォローする |
| フォロワー | Follower | 自分をフォローしているユーザー | ユーザーAのフォロワーはユーザーBとC |
| フォロー中 | Following | 自分がフォローしているユーザー | ユーザーAのフォロー中はユーザーBとC |
| タイムライン | Timeline | フォローしているユーザーの投稿を時系列で表示したもの | ホーム画面のタイムライン |
| いいね | Like | ユーザーが投稿に対して「いいね」をすること | ユーザーAが投稿1にいいねする |
| コメント | Comment | ユーザーが投稿に対してコメントをすること | ユーザーAが投稿1にコメントする |
| リポスト | Repost | ユーザーが他のユーザーの投稿を再投稿すること | ユーザーAが投稿1をリポストする |

#### 用語集の維持

- **定期的に更新する**: 新しい用語が追加されたら、すぐに用語集に追加
- **チーム全体で共有する**: 全員が同じ用語集を参照する
- **コードレビューで確認する**: 新しい用語が追加されたら、用語集に記載されているか確認

### ステップ3: コードでの使用

ユビキタス言語をコードで使用します。

#### 良い例: ユビキタス言語を使用

```python
# 用語集: 「投稿」= Post
class Post:
    def __init__(self, author, content):
        self.author = author
        self.content = content
    
    def publish(self):
        # 「投稿を公開する」という用語を使用
        pass

# 用語集: 「フォロー」= Follow
class User:
    def follow(self, other_user):
        # 「フォローする」という用語を使用
        pass

# 用語集: 「タイムライン」= Timeline
class Timeline:
    def get_posts(self):
        # 「タイムライン」という用語を使用
        pass
```

#### 悪い例: ユビキタス言語を使用しない

```python
# ❌ 悪い例: 技術的な用語を使用
class UserMessage:
    def send(self):
        # 「投稿」ではなく「メッセージ」という用語を使用（用語集と不一致）
        pass

# ❌ 悪い例: 曖昧な用語を使用
class Content:
    def create(self):
        # 「投稿」ではなく「コンテンツ」という曖昧な用語を使用
        pass
```

## ソーシャルメディアでのユビキタス言語の例

### コア概念

#### 1. User（ユーザー）

**定義**: システムを利用する人

**関連用語**:
- **Author（作成者）**: 投稿を作成したユーザー
- **Follower（フォロワー）**: 自分をフォローしているユーザー
- **Following（フォロー中）**: 自分がフォローしているユーザー

**コード例**:
```python
class User:
    def __init__(self, user_id, name):
        self.user_id = user_id
        self.name = name
        self._followers = set()  # フォロワー
        self._following = set()   # フォロー中
```

#### 2. Post（投稿）

**定義**: ユーザーが作成する公開コンテンツ

**関連用語**:
- **Content（内容）**: 投稿の本文
- **Author（作成者）**: 投稿を作成したユーザー
- **Created At（作成日時）**: 投稿が作成された日時

**コード例**:
```python
class Post:
    def __init__(self, post_id, author, content):
        self.post_id = post_id
        self.author = author      # 作成者
        self.content = content    # 内容
        self.created_at = datetime.now()  # 作成日時
```

#### 3. Follow（フォロー）

**定義**: ユーザーが他のユーザーをフォローすること

**関連用語**:
- **Follower（フォロワー）**: フォローする側のユーザー
- **Followee（フォロイー）**: フォローされる側のユーザー
- **Follow Relationship（フォロー関係）**: フォロー関係そのもの

**コード例**:
```python
class Follow:
    def __init__(self, follower, followee):
        self.follower = follower  # フォロワー
        self.followee = followee  # フォロイー
        self.created_at = datetime.now()
```

#### 4. Timeline（タイムライン）

**定義**: フォローしているユーザーの投稿を時系列で表示したもの

**関連用語**:
- **Home Timeline（ホームタイムライン）**: フォローしているユーザーの投稿
- **User Timeline（ユーザータイムライン）**: 特定のユーザーの投稿

**コード例**:
```python
class Timeline:
    def get_home_timeline(self, user):
        # ホームタイムラインを取得
        following = user.get_following()
        posts = self._get_posts_by_users(following)
        return sorted(posts, key=lambda p: p.created_at, reverse=True)
```

#### 5. Like（いいね）

**定義**: ユーザーが投稿に対して「いいね」をすること

**関連用語**:
- **Like Count（いいね数）**: 投稿に付けられたいいねの数
- **Liked By（いいねしたユーザー）**: いいねをしたユーザーのリスト

**コード例**:
```python
class Post:
    def __init__(self, post_id, author, content):
        self.post_id = post_id
        self.author = author
        self.content = content
        self._likes = set()  # いいねしたユーザーのID
    
    def like(self, user):
        # いいねする
        self._likes.add(user.user_id)
    
    def get_like_count(self):
        # いいね数を取得
        return len(self._likes)
```

#### 6. Comment（コメント）

**定義**: ユーザーが投稿に対してコメントをすること

**関連用語**:
- **Comment Author（コメント作成者）**: コメントを作成したユーザー
- **Comment Content（コメント内容）**: コメントの本文

**コード例**:
```python
class Comment:
    def __init__(self, comment_id, post, author, content):
        self.comment_id = comment_id
        self.post = post
        self.author = author      # コメント作成者
        self.content = content    # コメント内容
        self.created_at = datetime.now()
```

## ドメインエキスパートとの対話方法

### 対話の準備

1. **質問を準備する**: 曖昧な用語をリストアップ
2. **具体例を用意する**: 「この場合、何と呼びますか？」と具体例を示す
3. **記録を取る**: 対話の内容を記録し、用語集に反映

### 対話の実践

#### 例1: 「投稿」と「メッセージ」の違いを明確にする

```
開発者: 「ユーザーが作成するコンテンツについて質問があります。公開されるコンテンツと、ユーザー間で直接やり取りするコンテンツは、同じものですか？」

ドメインエキスパート: 「いいえ、違います。公開されるコンテンツは『投稿』、ユーザー間で直接やり取りするコンテンツは『メッセージ』と呼んでいます」

開発者: 「了解です。では、コードでも 'Post'（投稿）と'Message'（メッセージ）を別物として扱います。用語集に追加しますね」
```

#### 例2: 「フォロー」の詳細を明確にする

```
開発者: 「『フォロー』について質問があります。ユーザーAがユーザーBをフォローした場合、ユーザーBは自動的にユーザーAをフォローしますか？」

ドメインエキスパート: 「いいえ、自動的にはフォローしません。一方向の関係です」

開発者: 「了解です。では、'Follow'は一方向の関係として実装します。用語集に『一方向の関係』と記載しますね」
```

### 対話の継続

- **定期的に会議を開く**: 週1回程度、用語の確認と追加を行う
- **コードレビューで確認**: 新しい用語が追加されたら、用語集に記載されているか確認
- **ドキュメントを更新**: 用語集の変更があったら、関連ドキュメントも更新

## 用語集の作成と維持

### 用語集の形式

用語集は、以下の形式で作成します：

```markdown
# ソーシャルメディア 用語集

## コア概念

### Post（投稿）
- **定義**: ユーザーが作成する公開コンテンツ
- **関連用語**: Author（作成者）、Content（内容）、Created At（作成日時）
- **コードでの使用**: `class Post:`
- **例**: 「今日はいい天気です」という投稿

### Follow（フォロー）
- **定義**: ユーザーが他のユーザーをフォローすること（一方向の関係）
- **関連用語**: Follower（フォロワー）、Followee（フォロイー）
- **コードでの使用**: `user.follow(other_user)`
- **例**: ユーザーAがユーザーBをフォローする

## よくある質問

### Q: 「投稿」と「メッセージ」の違いは？
A: 「投稿」は公開コンテンツ、「メッセージ」はユーザー間で直接やり取りする非公開コンテンツです。

### Q: 「フォロー」は双方向ですか？
A: いいえ、一方向の関係です。ユーザーAがユーザーBをフォローしても、ユーザーBは自動的にユーザーAをフォローしません。
```

### 用語集の維持方法

1. **バージョン管理**: 用語集をGitで管理し、変更履歴を記録
2. **定期的なレビュー**: 月1回程度、用語集をレビューし、不要な用語を削除
3. **新メンバーへの共有**: 新メンバーが参加したら、用語集を共有

## ユビキタス言語の効果

### 効果1: コミュニケーションの改善

- ドメインエキスパートと開発者の間で用語が一致
- 会議の時間が短縮される
- 誤解が減る

### 効果2: コードの可読性向上

- コードからビジネスの意図が読み取れる
- 新メンバーが理解しやすい
- コードレビューが容易になる

### 効果3: 保守性の向上

- 用語が統一されているため、変更が容易
- ドキュメントとコードが一致
- テストコードも理解しやすい

## まとめ

- **ユビキタス言語**は、ドメインエキスパートと開発者が共有する共通言語
- **コード、ドキュメント、会議**のすべての場面で同じ用語を使用
- **用語集**を作成し、用語の定義を明確にする
- **ドメインエキスパートとの対話**を通じて、ユビキタス言語を確立する
