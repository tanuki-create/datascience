# 物体検出の基礎

物体検出は、画像の中に「何が、どこに、いくつあるか」を同時に推定するタスクです。分類が「画像全体に何が写っているか」を答えるのに対し、物体検出は各インスタンスの位置まで返します。セグメンテーションはさらに細かく、ピクセル単位で境界や領域を表現します。

この違いを最初に押さえると、以降の設計判断が整理しやすくなります。検出は「位置の誤差」「重複検出」「小物体」「クラス不均衡」「推論速度」が中心課題で、単なる分類モデルの延長ではありません。

## 物体検出で解く問題

- 入力: 画像または動画フレーム
- 出力: クラス名、confidence score、bbox
- 典型目的: 監視、製造検査、医療画像、リテール、ロボティクス、自動運転
- 実務上の制約: 精度だけでなく、レイテンシ、スループット、メモリ、運用容易性が重要

## 分類・セグメンテーションとの差

- 分類は「画像に写っている主対象」を当てる
- 検出は「各インスタンスの位置と種類」を当てる
- セグメンテーションは「どの画素が対象か」を当てる
- インスタンスセグメンテーションは検出と領域推定を合体したもの

検出は、分類よりもラベル設計とアノテーション品質の影響が大きく、セグメンテーションよりも推論系の最適化が効きやすい領域です。

## 代表的なモデル系統

### Two-stage 系

R-CNN, Fast R-CNN, Faster R-CNN の系統です。まず候補領域を作り、その後に各候補を分類・回帰します。

- 強み: 高精度、複雑な背景や難例に強い
- 弱み: 構成が重く、推論が遅め
- 向く場面: 精度優先、オフライン処理、医療や検査のように誤検出を抑えたい場面

### One-stage 系

SSD, RetinaNet, YOLO 系が代表です。候補生成と分類・回帰を一段で行います。

- 強み: 高速、実装と運用がしやすい
- 弱み: 小物体や密集シーンで難しくなりやすい
- 向く場面: リアルタイム推論、エッジデバイス、動画処理

### Transformer 系

DETR 以降の系列では、集合予測として検出を扱い、NMS への依存を減らします。

- 強み: 設計が一貫しており、ラベル割当を理論的に整理しやすい
- 弱み: 学習が重い、データ量やレシピに敏感なことがある
- 向く場面: 研究開発、長期的に保守しやすい構成を目指す場合

### Self-supervised backbone 活用

DINOv2 のような自己教師あり ViT をバックボーンに使うと、少量ラベルでも強い表現を得やすくなります。特に、転移学習の初期段階で効きやすいです。

- 強み: 表現学習が強く、少データやドメイン転移で有利
- 弱み: 検出ヘッドや解像度設計との相性を丁寧に調整する必要がある
- 向く場面: ラベルが高価、対象が多様、既存分類事前学習だけでは不足する場合

## 2024-2026 の重要アップデート

近年の物体検出は、単に AP を伸ばす競争から、次の 4 方向に分かれてきています。

### 1. real-time でも end-to-end 化を進める流れ

YOLOv10 は NMS-free な real-time 検出を整理し、RT-DETRv2 は real-time DETR を bag-of-freebies で改善しました。ここで重要なのは、「end-to-end = 常に現場で最適」ではないことです。最新研究は、後処理を減らす方向へ進んでいますが、実務では exporter、量子化、監視、閾値運用まで含めた総コストで評価する必要があります。

### 2. open-vocabulary detection の実用化

Grounding DINO や YOLO-World の登場で、事前に固定したクラス集合だけでなく、テキスト入力で対象を切り替える検出器が実用域に入りました。これにより、初期ラベリング、長尾クラス探索、PoC 立ち上げの速度は大きく上がりました。

一方で、SLA が厳しい本番推論では、依然として closed-set の専用 fine-tune モデルが優勢です。open-vocabulary detector は、最終製品そのものというより、データブートストラップや long-tail triage に強い道具として考える方が現実的です。

### 3. foundation backbone の浸透

DINOv2 が示したのは、「大規模で curated なデータ上の自己教師あり学習が、汎用視覚特徴を本当に実用化し始めた」という点です。これにより、少量ラベル、ドメイン転移、複雑背景での初期立ち上がりが改善しやすくなりました。

ただし、backbone が強くても、検出器全体が自動で強くなるわけではありません。実務では、neck、多段特徴、patch 粒度、fine-tuning 深度まで含めて設計し直す必要があります。

### 4. 「研究の最先端」と「本番の標準」が分離してきた

研究最前線では、open-vocabulary、multimodal grounding、vision foundation model 統合が主戦場です。一方、本番の標準は依然として、高速な closed-set detector を自社データに合わせて丁寧に回す形が多いです。プロとして重要なのは、この 2 層を混同しないことです。

## データセット

代表例は COCO, Pascal VOC, Open Images, LVIS です。実務では公開データセットよりも、自社データの分布とアノテーション規約の方が支配的になります。

- COCO: 汎用ベンチマークとして最重要
- VOC: 入門と比較の基準として有用
- Open Images: クラス数が多く、現実の長尾性を意識しやすい
- LVIS: long-tail を強く意識する評価に向く

2024 以降の研究を読むときは、COCO だけでなく、LVIS や ODinW のような長尾・open-world 的な設定も確認するとよいです。COCO で強いことと、現場の long-tail や promptable detection に強いことは一致しません。

データセットを見るときは、クラス数だけでなく、物体サイズ分布、遮蔽率、重なり、撮影条件、ラベルの粒度を確認します。

## 評価指標

- IoU: 予測 bbox と正解 bbox の重なり具合
- precision / recall: 誤検出と見逃しのバランス
- AP / mAP: クラスごとの平均適合率、検出研究の標準指標
- AR: どれだけ見逃しにくいかを見る補助指標
- FPS / latency: 実運用で重要

研究では mAP が中心ですが、運用では false positive と false negative のコスト構造を別に考える必要があります。

## 失敗パターン

- 小物体が消える
- 密集シーンで bbox が重なり、NMS で落ちる
- クラス不均衡で少数クラスが学習されない
- アノテーションの揺れで上限精度が伸びない
- ドメインシフトで本番精度が急落する
- confidence が過剰に高く、校正不良になる

## 学習の全体像

1. 問題定義を決める
2. ラベル設計とアノテーション基準を固定する
3. train / val / test を分布崩れなく分ける
4. ベースラインを作る
5. Augmentation と解像度を調整する
6. 評価指標を用途に合わせて決める
7. 誤検出を解析し、データと設定を更新する
8. 推論最適化まで含めて完成させる

物体検出は、モデルだけで完結しません。データ、ラベル、学習レシピ、評価、推論最適化が一体になって初めて性能が出ます。

## プロが最初に分解する 7 つの問い

実務で強い人は、モデル名より先に次の問いを切ります。

1. 何を検知対象にし、何を非対象にするのか
2. 見逃しと誤検知のどちらが高コストか
3. 1枚ごとの判定でよいのか、時系列の安定性が必要か
4. 小物体、遮蔽、密集、長尾のどれが支配的か
5. 学習データと本番データの差は何か
6. レイテンシ、スループット、メモリ制約はどこにあるか
7. 出力はアラート、ランキング、候補提示のどれとして使うのか

この 7 問が曖昧なまま進むと、どのモデルを選んでも改善が散発的になります。

## 問題設定でよくある失敗

- クラス定義が粗すぎて、後から運用で使い分けられない
- クラス定義が細かすぎて、ラベルが揺れて学習が壊れる
- 単一画像の検出問題だと思っていたが、実際には時系列判定が必要
- bbox で十分な問題に、最初からセグメンテーションを持ち込む
- 逆に、厳密な境界が必要なのに検出で済ませようとする

## データ中心で見るためのフレーム

物体検出では、モデルを変える前にデータを次の軸で棚卸しします。

- サイズ分布: 小物体、中物体、大物体の比率
- 視点分布: 正面、斜め、俯瞰、遠景の比率
- 環境分布: 昼夜、屋内外、天候、季節
- 難例分布: 遮蔽、反射、密集、動体ブレ、暗所
- ラベル分布: クラス頻度、欠損率、曖昧ラベル率

この棚卸しを言語化できる人ほど、改善施策が外れにくくなります。

## この分野の専門家がみな共有している、5つの中核的なメンタルモデル

- 検出は「分類 + 位置回帰 + 重複整理」の合成問題である
- モデル性能の多くは、ネットワークよりデータ分布で決まる
- 小物体、遮蔽、密集は別問題として扱うべきである
- 学習時の指標と運用時の指標は一致しないことが多い
- バックボーン、ヘッド、ラベル割当、NMS は独立に調整すべき部品である

## この分野の専門家たちが根本的に意見を異にしている点

### 1. Two-stage か One-stage か

立場A: 精度優先なら two-stage が堅い。難例と複雑シーンでの再現性が高い、というのが最も強い主張です。

立場B: 実運用では one-stage が現実的。速度、単純さ、配備容易性まで含めた総合点が高い、というのが最も強い主張です。

### 2. 検出を NMS ベースで扱うか、集合予測で扱うか

立場A: NMS は経験的に強く、実務で壊れにくい。レシピの自由度が高い、というのが最も強い主張です。

立場B: NMS は後処理の複雑さと不安定さを持ち込む。最初から集合予測として定式化した方が設計がきれい、というのが最も強い主張です。

### 3. 事前学習は分類事前学習で十分か、自己教師あり ViT が必要か

立場A: ImageNet 系の分類事前学習で十分な場面が多い。移植のしやすさと実績が最大の強み、というのが最も強い主張です。

立場B: DINOv2 のような自己教師あり表現は、少データやドメイン転移で本質的に強い。ラベルに依存しない表現の厚みが最大の強み、というのが最も強い主張です。

## 実務へのつなぎ方

最初は公開モデルの再現から始め、次に自社データへ移植し、最後に誤検出のパターン別にデータを増やします。検出は「学習を回した回数」よりも、「失敗例をどれだけ言語化して潰したか」で伸びます。

## 2026時点のプロ向け実務判断

- リアルタイム本番系:
  - まずは実績のある YOLO 系や RT-DETR 系でベースラインを作る
  - NMS-free は魅力的だが、export と profiling まで通して比較する
- 少量ラベル / 新規ドメイン:
  - DINOv2 系バックボーンや strong pretraining を優先検討する
- 長尾 / 未知クラス / データ立ち上げ:
  - Grounding DINO や YOLO-World を annotation bootstrap に使う
- 厳格運用:
  - 最終推論器は closed-set detector に落とし込み、閾値と責任分界を固定する

## 演習課題

1. 分類で十分な問題と、検出でなければ解けない問題を 5 つずつ挙げる
2. 同じ対象に対して、bbox、インスタンスセグメンテーション、キーポイントのどれを選ぶべきか比較する
3. ある案件で mAP より recall が重要になる理由を業務フロー込みで説明する
4. 小物体が支配的な案件で、入力解像度、アノテーション、カメラ配置のどれを先に直すべきか議論する

## セルフチェック

- one-stage と two-stage を、速度ではなく失敗モードの違いで説明できるか
- DINOv2 のような事前学習バックボーンが効く条件を 3 つ言えるか
- mAP が高くても導入失敗する理由を 3 つ説明できるか
- いま扱っている案件で、最も高コストな誤りが何か説明できるか
