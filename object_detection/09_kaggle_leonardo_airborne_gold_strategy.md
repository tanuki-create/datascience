# Kaggle: Leonardo - Airborne Object Recognition Challenge（ゴールド級攻略）

この文書は、**Leonardo - Airborne Object Recognition Challenge** を「Leaderboard の上位争い（メダル圏）」と **Efficiency Prize** の両方を視野に入れたときの、実戦向けの詳細戦略です。公開情報（Overview / Evaluation / Submission / Code Requirements / Efficiency Prize）を前提に、データ特性（機載・多条件・可視＋赤外など）に応じた設計判断を中心に書いています。

> 一次ソース: [Leonardo - Airborne Object Recognition Challenge](https://www.kaggle.com/competitions/leonardo-airborne-object-recognition-challenge)（Kaggle、2026）

---

## 0. 5 つの中核メンタルモデル（このコンペで勝つ順）

1. **指標は VOC mAP @ IoU 0.5 単一** — mAP50-95 や COCO 流の話ではない。閾値設計・NMS・キャリブレーションは「50 の精密さ」に最適化する。
2. **提出形式が特殊** — bbox は **相対座標 (0.0–1.0)**。空画像は **`None 1 -1 -1 -1 -1` の固定文字列**。後処理バグが即死する。
3. **ドメインは「きれいな監視カメラ」ではない** — ブラー、動揺、ズーム、視点変化、遮蔽、微小物体、多スペクトル。強い aug と推論時のマルチスケールが主戦場になりやすい。
4. **Code Competition** — **Notebook 提出・9 時間制・インターネット無効**。外部データは「フリー公開」なら可だが、**実行時にネットから落とせない**。重い事前学習は **Kaggle Dataset 化**が前提。
5. **Efficiency Prize は「精度と時間の合成」** — リーダーボード上位（または自動選定条件）かつ **sample より Private で良い**など条件あり。**mAP とランタイムのトレードオフが設計の一部**になる。

## 1. コンペ要約（確認用チェックリスト）

| 項目 | 内容 |
|------|------|
| タスク | 物体検出 + **クラス分類**（複数目あり得る） |
| 学習データ | 実機載映像由来、**可視 + 赤外** など複センサが言及されている |
| 評価 | **PASCAL VOC 2010 流の mAP、IoU > 0.5** |
| 提出 | `submission.csv`、`ImageId,PredictionString`、1 画像 1 行 |

| 空画像 | `None 1 -1 -1 -1 -1` |
| ランタイム | **CPU / GPU どちらも Notebook ≤ 9 時間** |
| ネット | **無効** |
| 事前学習 | **公開 pretrained は利用可**（ただしオフラインロード要） |

Efficiency Prize のスコアは「mAP とベンチマーク・最大スコア・**評価秒数**」から定義される（式は公式説明に従う）。**最終週は「速い提出」と「強い提出」を分けて持つ**のが現実的です。

## 2. ゴールドを狙う戦略の骨格

### 2.1 二正面作戦（Leaderboard 用 / Efficiency 用）

- **A 本隊（LB）**: 大きい入力解像度、強い backbone、マルチスケール推論、TTA（軽め）、モデルアンサンブル（2 モデルまで等）、pseudo-label は慎重に。
- **B 効率（Efficiency）**: 中型 backbone、単一スケール＋軽い multi-scale のどちらか一方、NMS / 閾値は探索済みの固定、FP16/TF32、バッチ推論、不要な可視化を削減。

同じ Notebook 内でフラグ切替えにすると、提出ミスが減ります。

### 2.2 データ理解が精度の 7 割

公式データが公開されたら **最初の 48 時間**で次を数値化します。

- **クラス長尾**（出現頻度、画像あたりインスタンス数）
- **解像度分布**（短辺・長辺・アスペクト比）
- **可視 vs IR のペア有無**（同じシーンの別モダリティか、完全別か）
- **ブラー・モーションブラーの有無**（EXIF は期待しない。エッジエネルギーやラプラシアン分布で粗視化）
- **物体ピクセル面積の分布**（小物体比率）
- **オクルージョン・画面端切り出しの割合**

ここが分かると、「DETR 系が効くか / YOLO 系で十分か」「マルチスケール必須か」が早い段階で決まります。

---

## 3. 検証設計（CV）— リークを殺す

機載データは **同一機体・同一日・同一コース** で連続フレームがまとまっていることが多いです。

- **推奨**: **グループ分割（GroupKFold）**  
  - グループキー候補: `video_id` > `flight_id` > `session_id` > （無ければ）時系列に近いメタデータ
- **禁止に近い**: ランダムに画像単位で割る（ほぼ同一シーンが train/val に跨ぐ）
- **疑似ラベル**を使うなら、**val は常にクリーン**に保ち、CV 全体の一貫性を崩さない

ローカル指標は **公式に合わせ VOC mAP@0.5** を再利用可能な実装で固定し、LB と相関を見ます（COCO eval をそのまま流用すると閾値感覚がズレる）。

---

## 4. モデル選定（2026 時点の現実解）

制約は **9h・オフライン・機載向け軽量の文脈**です。

### 4.1 第一候補（実装速度とコミュニティ資産）

| 系統 | 長所 | 注意 |
|------|------|------|
| **YOLO 系（Ultralytics 等）** | 学習・export・速い反復、小物体は解像度と data で拾いやすい | NMS・閾値チューニングが mAP50 の主戦場 |
| **RT-DETR / RF-DETR 系** | GPU 上で end-to-end が強いことがある、NMS 依存が薄い | 訓練コスト・実装の重さ、9h に収まるか要見積り |
| **DINO / Co-DINO 系（MMDetection）** | 難データで強いことが多い | 設定と依存が重め、提出パイプラインの簡潔さと相談 |

**実戦のおすすめ**: まず **単一フレーム検出で強い YOLO系 or RT-DETR 系でベースライン**を固め、CV で上振れがある方に寄せる。動画時系列は **後述の追加入力として段階的に**。

### 4.2 バックボーン

- **ImageNet 事前学習**は当然候補。
- データが「埋め込み空間の頑健さ」を要求するなら **DINOv2 系を backbone にした検出**も候補（実装コストと 9h の兼ね合い）。
- IR と RGB が **チャネル構造が違う**場合、**2 ストリーム late fusion** は強いが重い。**まずは単一モデル＋モダリティをチャネル or 条件付き aug** で様子見が安全。

## 5. 学習：ドメインに効く augmentation の優先順位

優先度の高いものから。

1. **モーションブラー / デフォーカス**（カーネル可変、方向ランダム）
2. **ジッター・カメラ震えシミュレート**（アフィン＋軽い elastic は控えめに）
3. **ランダムリサイズ＋アスペクト比の現実的範囲**（極端な伸びは機載では不自然になりがち）
4. **HSV / 彩度 / コントラスト**（昼夜・大気）
5. **Cutout / Mosaic / Copy-Paste** — クラス不均衡・稀少クラスがあるなら **Copy-Paste 相当が効きやすい**（実装とラベル整合に注意）
6. **MixUp 系** — 検出では効く場合と壊す場合がある。**CV で必ず比較**

**赤外が別チャネルでない**（単なる「別画像」）なら、**モダリティ間での正規化統計**を分けて記録し、混入ミスを防ぐ。

---

## 6. 損失・サンプリング・長尾

- **cls ロス加重**（class weight / focal の効き）を CV で確認。
- **画像レベルの class balance sampler** は長尾に効くことが多い。
- **異常に大きい bbox** や **極小 bbox** はアノテーション・座標系バグの疑い。EDA で除外ルールを決める。

---

## 7. 推論：mAP50 を取りにいく後処理

### 7.1 マルチスケール（MS）

- **短辺スイープ**（例: 2–3 スケール）で box を merge。重いほど LB は伸びやすいが Efficiency は不利。
- **TTA（水平反転のみ）**はコスパが良いことが多い。

### 7.2 NMS / Soft-NMS

- mAP50 では **Classic NMS ＋ IoU 0.5 付近の閾値探索**が基本。
- 擁壁が多いなら Soft-NMS を試すが、**提出形式への影響（スコアと box 数）**を必ず確認。

### 7.3 閾値とキャリブレーション

- クラス別閾値は **Platt / isotonic** よりまず **グリッドサーチ**で十分なことが多い。
- **confidence の較正**で mAP が伸びるケースあり（val で決め、test に持ち込む）。

### 7.4 空検出（None 行）

- 「物体なし」予測は、**max score < 閾値**なら公式通り `None 1 -1 -1 -1 -1`。
- 閾値が高すぎると **recall 喪失**、低すぎると **FP で mAP 低下**。**画像単位の objectness** を別途学習するのは上級策。

## 8. submission.csv の実装メモ（バグゼロ）

```text

def456,Obstacle 0.5 0.1 0.2 0.3 0.4 Drone 0.7 0.2 0.4 0.6 0.8
```

- `x_min < x_max`、`y_min < y_max`、範囲クリップ。
- クラス名は **訓練ラベルと完全一致**（大文字小文字・スペース）。
- **テスト全行・重複なし・ヘッダ必須**。

## 9. Code Competition オペレーション

1. **依存の完全オフライン化**  
   - `pip install` 対象を Notebook 冒頭で固定し、**ホイールを Dataset に同梱**する運用を検討。
2. **事前学習重み**は Kaggle Dataset として添付。`torch.hub` のオンライン取得に依存しない。
3. **再現性**  
   - seed 固定、`deterministic` 可能なら有効化（速度とトレードオフ）。
4. **9 時間監視**  
   - 学習 epoch を減らす代わりに **強い aug + 早期停止**、または **事前学習を厚くして fine-tune を短く**。
5. **提出用セル**は最後に **推論のみ**が走る構造にし、学習の再実行を避ける（コンペルールの範囲内で **学習済み weight を Dataset に載せる**戦略は定番だが、ルール改訂がないか毎回確認）。

## 10. Efficiency Prize に効く実装レバー

- **推論パイプラインの I/O**（画像読み込み、リサイズ）を **ベクトル化 / 並列化**。
- **CUDA 側への一括転送**、**AMP FP16**。
- **モデル融合は 2 個まで**等、上限を決める。
- **TTA はオフ**または反転のみ。
- **入力解像度を val で Pareto 最適点**に置く（mAP / 秒を散布図に）。

---

## 11. 疑似ラベル・外部データ（慎重に勝つ）

- 公開ルール上 **許可された外部データ**のみ。ドキュメント通り。
- 疑似ラベルは **高 conf のみ**、かつ **domain gap が大きいソースは有害**になりやすい。
- **student–teacher EMA** は強いが実装と時間が増える。CV で「本当に平均化 mAP が上がるか」を見る。

## 12. 動画情報の扱い（オプションの上位釘）

データがフレーム列なら、単純化の順に:

1. **現在フレームのみ**で最強まで持っていく  
2. **双方向の軽量な追跡（IoU tracker）**で box を安定化（スコア調整）  
3. **clip-level 特徴**はコスト高。Efficiency では最後の最後

### 17.0 この用語集の読み方（初学者）

1. **まず 17.1〜17.3** を読むと、後の用語が「突然の専門用語」にならなくなります。  
2. 各用語は **一言 → 詳しく →（ある場合）たとえ → このコンペで大事な理由 → つまずき** の順です。  
3. **図や数式で厳密に追いたい**場合は、本リポジトリの [02_terms_and_metrics.md](./02_terms_and_metrics.md) を併読すると安心です。

### 17.1 そもそも「物体検出」でプログラムは何をしている？

- **入力**: 画像 1 枚（ここでは飛行機などに載ったカメラの映像から切り出した 1 フレーム）。  
- **出力**: だいたい次の 3 つをセットにした **リスト**。  
  - **どこに**あるか … 長方形（**バウンディングボックス**）で示す  
  - **何が**写っているか … **クラス名**（例: ドローン、障害物など）  
  - **どのくらい自信があるか** … **confidence**（信頼度スコア、0 に近い〜1 に近いことが多い）  
- **学習の目的**: 人間が付けた正解の四角とクラスに、モデルの予測ができるだけ一致するよう、内部パラメータを少しずつ直していくこと。  
- **評価（このコンペ）**: 予測の四角が正解と **どのくらい重なったか（IoU）**、クラスは合っているか、などを踏まえて **PASCAL VOC 流の mAP（IoU 0.5）** で採点される。

**分類との違い（超重要）**: **分類**は「画像全体がネコかイヌか」のように **答えが 1 個**。**物体検出**は「同じ写真の中に複数の物体があり、それぞれ **場所付きで** 答える」仕事です。

### 17.2 バウンディングボックス（BBox）と座標 — 初学者が一番ハマる所

- **BBox（バウンディングボックス）**: 物体を囲む **長方形**。多くの実装では **左上の点**と**右下の点**の 2 点で表す（`x_min, y_min, x_max, y_max`）。  
- **ピクセル座標**: だいたい **画像の左上を (0,0)** とし、右に行くほど x が増え、下に行くほど y が増える。値は **整数**で数えることが多い。  
- **相対座標（このコンペの提出はこちら）**: 上の x,y を **画像の幅・高さで割って 0.0〜1.0** にしたもの。  
  - 例: 横 1000 画素の画像で、左から 100 画素の位置は `x = 0.1`（= 10% の位置）。  
  - **解像度が変わっても「割合」は同じ意味**なので、提出形式が揃えやすい。  
- **なぜ最初から相対にしないの？** ニューラルネットは計算の都合で、サイズを **640×640 など固定**にすることが多い。**学習・推論は縮小後、提出は原画像基準**の二段構えになりがちで、ここでバグが生まれやすい。

### 17.3 指標のつながり（TP / FP / FN → precision / recall → AP → mAP）— ざっくり版

厳密な定義は 02 に任せ、**感覚**だけここでつなぐ。

- **TP（真陽性）**: 本当に物体があるのに、**ちゃんと当てた**予測。  
- **FP（偽陽性）**: **実際には「当たり」と言えない**のに、あると予測したもの。  
  - 例: ずれた箱（IoU が足りない）、背景のゴミを物体と言った、同じ物体に箱が二重三重…など。  
- **FN（偽陰性）**: 本当はあるのに **見逃した**。  
- **precision（精度）**: ざっくり「**予測が“ある”と言ったもののうち、どれだけ本当だったか**」。予測箱を無闇に増やすと下がる。  
- **recall（再現率）**: ざっくり「**本当にある物体のうち、どれだけ拾えたか**」。閾値を上げて予測を捨てすぎると下がる。  
- **AP（Average Precision）**: **1 クラス**について、「閾値を変えたときの precision と recall の良し悪し」を 1 つの数値にまとめたもの（詳細は 02）。  
- **mAP（mean Average Precision）**: **クラスごとの AP を平均**したもの。このコンペでは **「IoU が 0.5 を超えたら位置は合格」** というルールのもとでの mAP が主役。

**初学者メモ**: mAP は **四角の位置**・**クラス**・**confidence と閾値**・**NMS** が全部絡むので、「モデルが強いのに mAP が低い」は **後処理のミス**であることが多いです。

### 17.4 コンペ・評価・提出まわり

#### Code Competition（コードコンペ）

- **一言**: 提出するのは「**Notebook を Kaggle が実行した結果**」。自分の PC の `.py` だけ送る形式ではない。  
- **詳しく**: 採点側が同じ手順で再実行し、`submission.csv` などを読む。利用可能時間・インターネット・保存先などがルールで決まる。  
- **たとえ**: レポートではなく「**再現手順付きの実験ノート**」を提出して、採点係に同じ実験をしてもらうイメージ。  
- **このコンペで**: **9 時間以内**に収めること、**ネットなし**で必要ファイルをすべて同梱すること、が最初の関門。ここで設計が決まる。  
- **つまずき**: ローカルでは動くのに Kaggle だけ落ちる → **パッケージのダウンロード**、**パス**、**GPU メモリ**、**重みファイルの場所**を疑う。

#### Leaderboard（リーダーボード, LB）

- **一言**: 提出ごとに付く **順位表**。  
- **詳しく**: **Public LB** は途中経過用に見える得点、**Private LB** は最終的に隠れたテストで決まる得点（コンペの説明に従う）。両者は同じ分布とは限らない。  
- **このコンペで**: チームのモチベーションと改善の方向付けに使うが、**LB だけ追うと過学習**しうる。  
- **つまずき**: Public は良いのに最後に順位が落ちる → **検証データの切り方（リーク）**や **Private との分布差**を疑う。

#### Public LB と Private LB

- **一言**: **順位計算に使うテスト画像のかたまりが違う**ことがある。  
- **詳しく**: どちらが「最終成績」かはルールで決まる。多くのコンペで **Private が最終**。  
- **このコンペで**: 「採点の最後の審判はどっちか」を必ず確認。気持ちの整理が付く。  
- **つまずき**: Public 用に阈值を焼き固めすぎて **一般化しない**。

#### Efficiency Prize（効率賞）

- **一言**: **精度だけでなく、評価にかかった時間など**も混ぜた効率スコアで争う副賞（数式・条件は**必ず公式**を読む）。  
- **詳しく**: 「とにかく重いモデル＋多重推論」は LB では強いが、効率部門では不利になりうる。  
- **このコンペで**: **Leaderboard 専用の大型設定**と、**効率賞狙いの軽量設定**を両方持てると戦略の幅が広がる。  
- **つまずき**: 「GPU 推論は速いはず」なのに遅い記録 → **画像読み込み・CPU 前処理・単枚ループ**が支配的かもしれない。

#### sample_submission / benchmark

- **一言**: 主催者が配る **正しい提出の見本**と、比較用のベースライン。  
- **詳しく**: 列名、区切り、**物体なしのときの文字列**、クラス名の綴りは **ここが基準**。  
- **このコンペで**: `None 1 -1 -1 -1 -1` の形は **自分で省略・変更しない**。  
- **つまずき**: 「confidence の 1 はダミーだから消そう」→ **形式エラー**の典型。

#### submission.csv

- **一言**: このコンペで採点器が読む **決められたファイル名**の CSV。  
- **詳しく**: たいてい `ImageId,PredictionString` のヘッダ付きで、**テスト画像 1 枚につき 1 行ちょうど**。  
- **このコンペで**: 名前を間違える・パスを間違えると **0 点付近**になりうる。出力セルは固定化する。  
- **つまずき**: 行数がテスト枚数と一致するか、`ImageId` の欠損・重複がないか。

#### PredictionString（予測文字列）

- **一言**: **1 枚分の予測を、CSV の 1 マスに文字として全部詰め込んだもの**。  
- **詳しく**:  
  - 1 物体: `クラス名 confidence x_min y_min x_max y_max`（スペース区切り）  
  - 複数物体: それを **同じセル内でスペースで続ける**（公式例に完全一致）  
  - 物体なし: サンプル通り（例: `None 1 -1 -1 -1 -1`）  
- **このコンペで**: コンマを混ぜた、順序を入れ替えた、クラス名を勝手に短縮した、など **1 か所の typo で無効**になりうる。  
- **つまずき**: 学習データの `Drone` と提出の `drone` が不一致、など **文字の揺れ**。

#### 相対座標（0.0–1.0）

- **一言**: 箱の位置を **画像サイズで割った 0〜1 の割合**で書く。  
- **詳しく**: **必ず** `x_min < x_max`, `y_min < y_max`。はみ出しを防ぐなら **[0,1] にクリップ**（ルールで禁止されていなければ）もよく使う。  
- **このコンペで**: **ピクセルのまま提出**すると採点不能や大失点。ネット入力用に縮小した座標のまま **戻し忘れ**も多い。  
- **つまずき**: 「リサイズ後の 640 基準で相対化」してしまい、**元画像基準の提出**とズレる。

#### PASCAL VOC / VOC mAP

- **一言**: 物体検出の評価の流儀のひとつ。**産業界・コンペでもまだよく出る**。  
- **詳しく**: **COCO** という別の有名評価とは、「どの IoU を当たりとするか」など細部が違う。同じ `mAP` でも中身が別ゲーム。  
- **このコンペで**: ローカルで **COCO 用の評価だけ**を見ながら閾値を決めると、**本番指標とズレる**。  
- **つまずき**: ライブラリのデフォルト評価を信じて、**別のルールで練習している**。

#### mAP@0.5（mAP50）

- **一言**: **IoU が 0.5 を超えたら「場所は合格」**とみなして計算される mAP。  
- **詳しく**: より厳しい 0.75 や 0.5〜0.95 平均などは、**このコンペの主指標の説明からは外れる**（主催者の最新記述を必ず確認）。  
- **このコンペで**: 少しズレた箱でも **0.5 を割ればアウト**、ギリ割ればセーフ。閾値・NMS の「勝ち方」が変わる。  
- **つまずき**: COCO 用に鍛えた習慣（高 IoU を狙う）だけでは **mAP50 最適ではない**。

#### IoU（Intersection over Union）

- **一言**: **2 つの BBox がどれだけ重なっているか**を表す 0〜1 の数。  
- **詳しく**:  
  \[
  \text{IoU} = \frac{\text{交差の面積}}{\text{結合の面積}}
  \]  
- **たとえ**: 卓上に **二枚の付箋**を貼ったとき、**重なった部分の面積 ÷ 二枚をまとめて囲む最小の長方形の面積**に近いイメージ。  
- **このコンペで**: **0.5 超え**が当たり判定の境界。**NMS** も「同じ物体か？」の目安に IoU を使う。  
- **つまずき**: 座標の **x/y の順**や **原画像 vs リサイズ後**を間違えると、IoU が全部低く出て **学習も評価も破壊**される。

#### confidence（信頼度スコア）

- **一言**: その予測を **どれだけ有力とみなすか**のスコア。たいてい **0〜1** に近いレンジで出す。  
- **詳しく**: 厳密には「確率」かどうかはモデル次第。**並べ替え用のスコア**として扱われることも多い。  
- **このコンペで**: **スコア閾値**を下げると予測が増え recall は上がりやすいが **FP も増える**。mAP は途中で頭打ちになったり下がったりする。  
- **つまずき**: 0〜100 の **百分率**で出していないか、別ライブラリの生の **logit** のまま出していないか。

### 17.5 データ・検証設計

#### EDA（探索的データ分析）

- **一言**: モデルを動かす前に **データの中身を眺めて分布を掴む**作業。  
- **詳しく**: 画素の明るさ、解像度、物体の大きさ、クラス頻度、ブラーっぽい画像の割合、アノテーションの変な外れ値など。  
- **このコンペで**: 機載は **小さい・暗い・ボケる**が出やすい。**何に効く工夫をするか**の優先順位がここで決まる。  
- **つまずき**: とりあえず大モデル → **データの穴**を見ていない。

#### CV / Cross-Validation（交差検証）

- **一言**: 手持ちの学習データを **何度か入れ替えながら**「本番っぽく性能を測る」手順。  
- **詳しく**: 例: 5 分割なら、5 通りの train/val の組み合わせを回してスコアを平均する。  
- **このコンペで**: LB だけだと **たまたま Public に合うチューニング**が起きる。CV はその抑制役。  
- **つまずき**: CV の切り方が甘いと **リーク**し、CV スコアだけ妙に高い。

#### GroupKFold（グループ付き K 分割）

- **一言**: 「**同じグループ**（同じ動画など）に属する画像は、**学習と検証に同時に出ない**」ように切る。  
- **詳しく**: ランダムに 1 枚ずつ割るより厳しいが、**実運用に近い難しさ**を測れる。  
- **このコンペで**: 連続フレームは **中身が似すぎ**。通常のランダム分割は **簡単すぎる検証**になりがち。  
- **つまずき**: メタデータに ID が無い → **ファイル名・時刻**から擬似グループを作る必要あり。

#### リーク（情報リーク）

- **一言**: 検証やテストに **本番では見えてはいけない情報**が混ざること。  
- **詳しく**: 極端例は不正。実務では **同一シーンの一部が val に混ざる**などの「薄いリーク」が多い。  
- **このコンペで**: CV は良いのに Private が崩れる **定番原因**。  
- **つまずき**: 「拡張でテストっぽい加工」をしただけではリークではないが、**分割ミス**ではよく起きる。

#### クラス長尾（クラス不均衡）

- **一言**: **よく出るクラスと、めったに出ないクラスが同居**している状態。  
- **詳しく**: モデルはデータの多い方に引っ張られ、稀クラスを「たぶん多い方のクラス」と誤りやすい。  
- **このコンペで**: mAP はクラス平均なので **尾部を放置すると全体が沈む**。sampler・loss・Copy-Paste などを検討する動機になる。  
- **つまずき**: 「全体の accuracy」だけ見ると長尾が見えない。**クラス別のミス**を必ず見る。

#### オクルージョン（遮蔽）

- **一言**: 物体が **他のものに隠れて一部しか見えない**状態。  
- **詳しく**: 機体の一部、雲、地上の構造物などで起きやすい。  
- **このコンペで**: 教科書的な「きれいな物体写真」より難しい。**Cutout** などで強くなることがある。  
- **つまずき**: 物体が小さい＝遠い、とは限らず **手前で切れている**だけ、を見誤る。

#### 可視 / IR（赤外）・モダリティ

- **一言**: **見える光**の画像と、**赤外**など別センサの画像は、同じ景色でも **見え方のルールが違う**。  
- **詳しく**: グレースケールを無理に 3 チャネル化したり、ImageNet の正規化をそのまま当てたりすると **分布が壊れる**ことがある。  
- **このコンペで**: 「RGB としての常識」を IR にそのまま適用していないか。**ペア学習できるか**で設計が変わる。  
- **つまずき**: IR を **ただのモノクロ JPG** と思い込み、前処理だけ RGB 前提にする。

### 17.6 モデルアーキテクチャ

#### backbone（バックボーン）

- **一言**: 画像から「特徴」という中間表現を取る **土台のネットワーク**。  
- **詳しく**: その後ろに **neck**（解像度をまとめる等）や **head**（どこに何があるかを出力）が付くことが多い。  
- **たとえ**: 料理で **だしを取る工程**。この後の head が味付け。  
- **このコンペで**: **大きいほど表現力は上がりやすいが遅い**。LB 用と Efficiency 用でサイズを分けるのが現実的。  
- **つまずき**: backbone だけ大型化して **入力解像度が小さいまま** → 小物体で伸びない。

#### neck / head（ネック / ヘッド）

- **一言**: **neck** はバックボーンの複数スケールの情報を束ねる。**head** は実際に **位置・クラス・confidence** を出す部品。  
- **詳しく**: 「CNN が全部」と思うと head の役割を忘れがち。  
- **このコンペで**: 小物体が重要なら **高解像度の情報が生きる neck** が効く場面がある。  
- **つまずき**: 特徴マップの可視化だけして、**head 直前の解像度とアンカー設計**を見ていない。

#### 事前学習（pretrained weights）

- **一言**: コンペのデータ **より遥かに大きいデータ**で、先に学習した重みからスタートすること。  
- **詳しく**: エッジやテクスチャなど **低〜中レベルの見え方**が最初から載っている。  
- **このコンペで**: **ゼロから学ばない**のが基本。ただし Code Competition では **ファイルとして読める形**が必要。  
- **つまずき**: 実行時に初めてネットから `wget` する設計。

#### ImageNet 事前学習

- **一言**: 一般物体の分類データセット（ImageNet）で学習した **バックボーンの初期値**。  
- **詳しく**: ドメインは機載と違うが、まず **強いベースライン**になりやすい。  
- **このコンペで**: 最初の 1 本目はほぼこれでよい。  
- **つまずき**: ImageNet 用の **RGB 正規化を IR に機械適用**していないか。

#### DINOv2

- **一言**: **ラベルなしに近い方法**で鍛えた、汎用的な視覚用バックボーンの一系統。  
- **詳しく**: 「きれいにラベル付された画像」に依存しすぎない強い表現を狙う系統（詳細は論文・公式）。  
- **このコンペで**: 外観変化が激しいときに **効くことがある**。ただし **重く、導入コスト**もある。  
- **つまずき**: 9 時間制と Notebook の制約の **せめぎ合い**。

#### YOLO 系

- **一言**: **速く回せる 1-stage 検出器**の代表格。実装例・解説が豊富。  
- **詳しく**: だいたい **NMS** などの後処理がセット。  
- **このコンペで**: **実験の回転が速い**のが魅力。mAP50 では **NMS と閾値**が伸びしろになりやすい。  
- **つまずき**: 入力 **640 固定**のまま小物体を諦めていないか。

#### DETR 系

- **一言**: **Transformer** で画像から物体を **セット**として予測する流れの検出器。  
- **詳しく**: アンカーに依存しない等、理屈は分かりやすいが **学習が重い・収束にくい**こともあった（改良系で緩和）。  
- **このコンペで**: 重なり・難配置で **素質が出る**ことがある。**9h** とセットアップ量との相談。  
- **つまずき**: データが少ないのに **学習が足りずスコアが出ない**。

#### RT-DETR / RF-DETR

- **一言**: DETR の流れを **リアルタイム寄り**にした系統。  
- **詳しく**: 実装・依存は都度確認。  
- **このコンペで**: **LB と Efficiency の中間**を狙う候補になりうる。  
- **つまずき**: 環境構築だけで **1 日溶ける**パターン。

#### DINO / Co-DINO（検出器の DINO）

- **一言**: DETR 系の高性能ライン（**DINOv2 とは別物**なので名前に注意）。  
- **詳しく**: MMDetection 上で試しやすいが設定が多い。  
- **このコンペで**: **難データで一発上振れ**があり得る反面、**提出 Notebook が複雑化**。  
- **つまずき**: 論文スコアを信じて **提出期限内に回らない**。

#### MMDetection / Ultralytics

- **一言**: 物体検出のための **実装コレクション＋学習の枠組み**。  
- **詳しく**: ざっくり、MMDetection は config が細かく研究寄り、Ultralytics は手軽で反復が速い、という使い分けの人が多い（個人差あり）。  
- **このコンペで**: **9h で確実に回る方**を選ぶのが賢い。  
- **つまずき**: バージョンアップで **デフォルト aug が変わり再現性が飛ぶ**。

#### late fusion（遅延融合）

- **一言**: 可視用と IR 用など **別々に特徴を取ってから後で合体**。  
- **詳しく**: 精度は上がりやすいが **だいたい遅い**。  
- **このコンペで**: LB 専用の重い選択肢になりがち。効率部門では諦めるか軽量化。  
- **つまずき**: **片方の前処理だけバグ**があると、合体しても信じられない出力に。

#### End-to-end（エンドツーエンド）

- **一言**: 人の手による複雑な後段ルールを減らし、**学習で一続きに寄せる**設計思想。  
- **詳しく**: NMS-free 的な扱いになることもあるが万能ではない。  
- **このコンペで**: 結局 **VOC mAP@0.5 で測られる**ので、手元評価と相性確認が必須。  
- **つまずき**: 学習は終わったが **推論マージが手作業だらけ**でデバッグ不能に。

#### アンサンブル

- **一言**: **複数モデルの答えを合体**して 1 つにする。  
- **詳しく**: 平均、投票、box を束ねる WBF など手法は様々。  
- **このコンペで**: **精度は伸びやすいが時間は増える**。効率提出では省略や薄化が普通。  
- **つまずき**: 合体後に **提出形式・クラス名・座標**が壊れる。必ずテスト。

### 17.7 学習・データ拡張・最適化

#### augmentation（データ拡張）

- **一言**: 学習中に **画像と枠をいじってバリエーションを増やす**。  
- **詳しく**: 反転、明るさ、ぼかし、リサイズ、貼り付け、など。  
- **このコンペで**: 実機の厳しい見え方に **先に慣れさせる**のが主目的。  
- **つまずき**: 強すぎて **ラベルと画像が矛盾**（見えないのに bbox だけ残る）。

#### モーションブラー / デフォーカス

- **一言**: **動いてボケた・ピンが合っていない**見え方を真似るぼかし。  
- **詳しく**: 方向や強さをランダムにすることが多い。  
- **このコンペで**: 実データに多い。**見た目は汚いが強くなる**典型。  
- **つまずき**: 強すぎて **小物体が消滅**。

#### Mosaic / Copy-Paste / Cutout / MixUp

- **一言**: タイル状に並べる・物体をコピーする・穴を開ける・画像を混ぜる、などの **拡張の束**。  
- **詳しく**: データが少ない・尾部が辛いときに効くことがある。**bbox の整合**を壊しやすい。  
- **このコンペで**: 稀少クラス対策の本命候補。**CV で必ず効くか確認**。  
- **つまずき**: Mosaic で **bbox が画像外だらけ**になっていないか。

#### cls ロス（分類ロス）

- **一言**: **「何クラスか」を間違えたらペナルティ**を与える部分。  
- **詳しく**: 検出では候補（アンカーやクエリ）ごとに付く。  
- **このコンペで**: 似た見た目のクラス混同は **mAP を静かに削る**。混同行列を見る。  
- **つまずき**: localization は悪くないのに **クラスだけ崩壊**。

#### focal loss

- **一言**: **簡単なサンプルばかり学習が支配する**のを抑える分類ロスの工夫の一つ。  
- **詳しく**: RetinaNet で広く知られた。常に勝つわけではない。  
- **このコンペで**: 背景が難しい、不均衡が強いときに試す価値。  
- **つまずき**: 「入れたら強いはず」だけで **閾値調整を怠る**。

#### class balance sampler

- **一言**: **ミニバッチの作り方**を工夫して、尾部クラスが乗りやすくする仕組み。  
- **詳しく**: 実装ごとに挙動が違う。  
- **このコンペで**: **尾部 recall**改善で mAP 底上げの可能性。  
- **つまずき**: 同じ画像ばかり繰り返し **過学習**。

#### 疑似ラベル（pseudo-label）

- **一言**: モデルが付けたラベルを、**追加の学習データとして再利用**する。  
- **詳しく**: 誤ラベル（ノイズ）が混ざる。**高 confidence だけ**など制限が一般的。  
- **このコンペで**: ドメインが近い追加画像に効きやすい。遠いドメインは逆効果も。  
- **つまずき**: 間違いを学習で固定する **confirmation bias**。

#### student–teacher EMA

- **一言**: **ゆっくり更新される教師**が、**生徒モデル**を安定した方向へ引っ張る半教師ありの型。  
- **詳しく**: 教師の重みを過去の指数移動平均（EMA）で滑らかに保つ等。  
- **このコンペで**: 未使用画像の活用に強いが **実装と時間**が増える。  
- **つまずき**: teacher が弱いと **誤学習のループ**。

#### 早期停止（early stopping）

- **一言**: 検証スコアが伸びないなら **学習を途中で止める**。  
- **詳しく**: 過学習防止と時間節約。  
- **このコンペで**: **9 時間**の貴重な枠を無駄にしない安全装置。  
- **つまずき**: 指標のノイズで早切り → **patience** 調整。

#### AMP / FP16 / TF32

- **一言**: **bit 幅を落として計算を速くする**系の総称。  
- **詳しく**: FP16 は半精度、AMP はそれを自動混在、TF32 は NVIDIA GPU の演算モードの話、など。  
- **このコンペで**: **Efficiency の秒数削減**に直結。ただし NaN や再現性の議論も出る。  
- **つまずき**: loss が NaN → **学習率や aug**とセットで疑う。

### 17.8 推論・後処理

#### マルチスケール（MS）推論

- **一言**: 同じ画像を **いくつかのサイズ**で走らせ、結果をまとめる。  
- **詳しく**: 遠い小物体は **拡大入力**が効く、という典型的な動機。  
- **このコンペで**: LB では伸びやすい。**時間は倍増しやすい**。  
- **つまずき**: マージ時に **元画像座標へ戻す**のを間違える。

#### 短辺スイープ

- **一言**: 入力をリサイズするとき **短辺の長さをいくつか試す**やり方。  
- **詳しく**: アスペクト比を保つリサイズが一般的。  
- **このコンペで**: MS の具体的手。**何ピクセルを試すか**が精度と速度のノブ。  
- **つまずき**: 極端な横長で **情報が潰れる** → タイル推論も検討余地。

#### TTA（Test-Time Augmentation）

- **一言**: 推論のときにも **左右反転など変形を掛けて**予測を統合する。  
- **詳しく**: 学習時より軽い変形にすることが多い。  
- **このコンペで**: **左右反転だけ**はコスパが良いことが多い。多重 TTA は効率の敵。  
- **つまずき**: bbox を元画像に戻す **逆変換ミス**。

#### NMS（Non-Maximum Suppression）

- **一言**: **同じ物体に対する重複した箱を間引く**。  
- **詳しく**: 「一番スコアが高い箱を残し、それと IoU が高い他の箱を消す」を繰り返すのが典型。  
- **このコンペで**: **NMS の IoU 閾値**は mAP50 に効きやすい。**グリッドサーチ対象**。  
- **つまずき**: NMS の前後で **座標のスケールが食い違う**。

#### Soft-NMS

- **一言**: 重複箱を **即削除ではなくスコアを下げる** NMS。  
- **詳しく**: 密集した正解に強いことがある。  
- **このコンペで**: 試す価値はあるが **FP 増**にもなり得る。  
- **つまずき**: 低スコアの箱が大量について **mAP 逆効果**。

#### 閾値（スコア閾値）

- **一言**: **confidence がこの値未満は予測として捨てる**という線。  
- **詳しく**: **クラス別**に別々の閾値を持つと伸びることがある。  
- **このコンペで**: **見逃しとゴミ予測のトレードオフ**を直接いじるノブ。  
- **つまずき**: val で最適でも test でズレることはあるが、**無いよりマシ**なことが多い。

#### キャリブレーション / 較正

- **一言**: モデルが言う「70%」が、**だいたい 7 割当たる**ように直すイメージ。  
- **詳しく**: スコアは並べ替えに使えるが、確率として歪んでいることがある。  
- **このコンペで**: 閾値最適の効き方が変わり **mAP が動く**ことがある。  
- **つまずき**: 「テストの正解は見えない」ので、較正は **ホールドアウトした検証**だけで行う。

#### Platt scaling / isotonic regression

- **一言**: **スコアの後から単調な変換**を当てはめて「確率っぽく」直す代表的手法。  
- **詳しく**: データが少ないと過学習しやすい面もある。  
- **このコンペで**: グリッドサーチが足りないときの **おまけ枠**。  
- **つまずき**: val が小さいと **壊す**ことも。

#### グリッドサーチ

- **一言**: 候補の数値を **並べて総当たり**し、一番良いのを選ぶ素朴だが強い方法。  
- **詳しく**: NMS 閾値、スコア閾値の探索に最適。  
- **このコンペで**: **短時間で効く**ことが多い、**最初にやるべき後処理**。  
- **つまずき**: val に **過学習した閾値** → 別 fold でも確認。

#### recall / FP（偽陽性）

- **一言**: recall は **見逃しを減らす方向**、FP は **ないのにあると言うミス**。  
- **詳しく**: mAP はこのバランスの上に成り立つ。  
- **このコンペで**: **空画像の None 行**とセットで考えると頭が整理される。  
- **つまずき**: 目視では「たくさん当てている」ように見えて **mAP 的には FP 地獄**。

#### objectness

- **一言**: 「**そもそもここに物体があるか**」のスコアや専用のヘッド。  
- **詳しく**: アーキテクチャにより明示的・暗示的。  
- **このコンペで**: **空画像**をきれいに扱いたい上級者向けのヒント。  
- **つまずき**: 実装コストに見合うか **数値でしか判断しない**。

#### IoU tracker

- **一言**: **前後フレームで IoU が高い箱同士をつなぎ**、同じ物体として追う軽い追跡。  
- **詳しく**: 深いトラッカーより単純で速い。  
- **このコンペで**: 動画なら **ちらつき低減**。単体画像評価なら保険。  
- **つまずき**: 速い運動で **ID が入れ替わる**。

### 17.9 Kaggle・実務・その他

#### Kaggle Dataset

- **一言**: 自分でアップロードしたファイル束を、Notebook から **マウントして読む**仕組み。  
- **詳しく**: 事前学習済み重み、追加データ、`.whl` などを載せられる。  
- **このコンペで**: **ネットなし**でもファイルを持ち込む本命。**ルールで許されるか**は毎回確認。  
- **つまずき**: ライセンス違反・規約違反データ。

#### オフライン化

- **一言**: 実行中に **インターネットに頼らない**ように依存を固めること。  
- **詳しく**: `pip install` もネットが要るので、ビルド済みパッケージ同梱などが必要になる。  
- **このコンペで**: **torch.hub の初回 DL** などに依存していないか点検。  
- **つまずき**: 「ローカルでは conda が勝手に取ってきた」＝ Kaggle では **再現不能**。

#### ベクトル化 / 並列 I/O

- **一言**: **for 文で 1 枚ずつ**より、**まとめて読んで・変換して・GPU に送る**。  
- **このコンペで**: GPU は速いうえに **CPU が詰まって時間が伸びる**ことがある。  
- **つまずき**: DataLoader の `num_workers`、**zip の展開コスト**。

#### Pareto 最適（フロンティア）

- **一言**: **精度と速度を同時に無限に最大化はできない**ときの「取り得る良いバランスの境目」。  
- **詳しく**: 横軸時間・縦軸 mAP で点を打つと、外側の線上がパレートフロンティア。  
- **このコンペで**: **1 個の設定で LB と効率賞を両取り**しようとすると破綻しやすい。  
- **つまずき**: 「最強モデル一択」で **副賞を捨てる**。

#### baseline（ベースライン）

- **一言**: **最初に超えるべき最低ライン**のスコア・実装。  
- **詳しく**: 複雑化する前に **単純な一打**を固定しておく。  
- **このコンペで**: 改良の効果測定のための **基準点**。  
- **つまずき**: 最初から複雑怪奇 → **デバッグ不能**。

#### epoch / バッチ（よく出る関連語）

- **epoch**: 学習データを **だいたい一通り見終わる**ラウンド。  
- **batch**: その途中の **小さなかたまり**。GPU メモリに載る単位。  
- **このコンペで**: 9h に収めるため **epoch を減らすが学習率スケジュールは別設計**、などの割り切りが出る。  
- **つまずき**: batch 1 で **BatchNorm が破綻**する等（freeze や別の正規化手法へ）。

#### 損失（loss）と最適化

- **一言**: **間違いを数値化**し、小さくなる方向にパラメータを動かす。  
- **詳しく**: optimizer（SGD, AdamW 等）が更新式を担う。  
- **このコンペで**: NaN は **aug・lr・数値精度**を疑う第一候補。  
- **つまずき**: bbox ロスと cls ロスの **スケールの扱い**のミス。

#### ドメインギャップ

- **一言**: **学習データの世界**と **本番データの世界**が違うこと。  
- **詳しく**: ImageNet は一般写真、機載は俯瞰・モーションブラー等。  
- **このコンペで**: だから **拡張とデータ設計**が効く、と説明できる語。  
- **つまずき**: 難しいドメイン適応より、**分割と現実的 aug**を先に。

#### リポジトリ内 02 / 04

- **一言**: 同じ `object_detection` フォルダの **用語の辞書**と**学習の実務本**。  
- **詳しく**: [02_terms_and_metrics.md](./02_terms_and_metrics.md)、[04_training_playbook.md](./04_training_playbook.md)。  
- **このコンペで**: この §17 は **読み物としての入口**。**数式レベル**は 02 で統一。  
- **つまずき**: 最初から完璧な厳密さを求めすぎる → **まず提出と再現**。
