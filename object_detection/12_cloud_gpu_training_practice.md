# クラウド GPU を使った学習 — プロ現場のやり方（最新情報の置き場）

物体検出・画像系の学習を **オンプレやローカルではなくクラウド GPU** で回すときの、**選び方・落とし穴・コスト・再現性**をまとめた実務メモです。料金・インスタンス型番・API は **クラウドの値上げ・新世代 GPU の追加が速い**ため、数値は **必ず各社の公式料金・クォータ画面**で確かめてください（本文は判断軸とパターンに寄せる）。

**位置づけ**: 本リポジトリの [04_training_playbook.md](./04_training_playbook.md)（学習設定）・[08_end_to_end_training_workflow.md](./08_end_to_end_training_workflow.md)（ツール）の **「どこで GPU を回すか」** を補う。

**更新目安**: 四半期ごとにインスタンス世代・マネージド製品名を見直す。

---

## 0. プロが最初に握る 5 つの軸

1. **コンピュートは借り物** — ディスクとプロセスはいつ消えてもよい前提。**チェックポイントとデータの置き場**を分離する。  
2. **コストは「GPU 時間 × 台数 × 実行回数」** — 失敗のたびに課金される。**小さく試してからスケール**が鉄則。  
3. **再現性は自分で運ぶ** — イメージ digest、依存の固定、`requirements.txt` / `uv.lock`、**Git のコミット**、シード。マネージドでも **コンテナ or env yaml** を版管理する。  
4. **クォータとリージョン** — 新規アカウントは **GPU 割当ゼロ**のことが多い。**事前申請・別リージョン**が必要。  
5. **データと権限** — 社外クラウドに上げる **画像の可否**（個人情報・図面の機密）。**VPC・PrivateLink・暗号化**を要件に含める。

## 1. なぜクラウド GPU か（現場の典型的動機）

| 動機 | コメント |
|------|----------|
| **ローカルに H100 が無い** | 短期だけ最新 GPU が欲しい。 |
| **並列で試したい** | ハイパラ探索・複数 seed を **複数ジョブ**に分割。 |
| **データがクラウドに既にある** | S3 / GCS からの転送コストと **学習リージョン**を揃える。 |
| **チームで共有** | キュー・優先度・予算アラートを組織で統一。 |

**向かない例**: 24/7 フル稼働の大規模学習だけを年単位で行うなら、**専用クラスタ or リース**のほうが安いことがある（TCO は財務・調達と計算）。

## 2. 形態の選び方（大枠）

### 2.1 IaaS: 自分で VM を借りる（EC2 / GCE / Azure VM 等）

- **長所**: 自由度が高い。SSH で入り **デバッグしやすい**。任意の Docker。  
- **短所**: **OS・ドライバ・CUDA の面倒**は自分。スケールアウトも自作。  
- **向き**: 検出の PoC、**カスタム CUDA**、既存オンプレに近い運用。

### 2.2 マネージド「学習ジョブ」（SageMaker Training / Vertex AI Custom Training / Azure ML Jobs 等）

- **長所**: **コンテナを投げるだけ**、ハイパラ探索・スケジューリング・メトリクス連携が揃いがち。企業の **監査・IAM** と相性が良い。  
- **短所**: ベンダーロックイン寄り。**デバッグはログ前提**。初回の **イメージビルド・権限**に時間がかかる。  
- **向き**: **本番パイプライン**、複数人で回すチーム、コンプライアンス重視。

### 2.3 サーバレス／従量秒課金（Modal 等）＋従来の GPU レンタル（RunPod / Lambda Labs 等）

- **長所**: **短時間ジョブ**に強い。インフラコードが短い。  
- **短所**: 組織によっては **セキュリティ審査**が必要。長時間大規模分散は **IaaS / マネージド**のほうが安定することが多い。  
- **向き**: **ファインチューン短時間**、研究スパイク、スタートアップの試行。

**プロの割り切り**: 「**まず 1 枚の VM で学習が完走**」→「**Spot + チェックポイント**でコスト削減」→「**ジョブ化してチーム展開**」の順が失敗しにくい。

## 3. GPU・インスタンスの選び方（物体検出の感覚）

一般論（**VRAM とバッチ・解像度**が支配的）:

| 規模の目安 | よく検討する GPU クラス（例） | メモ |
|------------|-------------------------------|------|
| **試行・小モデル** | T4 / L4 / A10G 系 | コスト優先。`batch` と `imgsz` を下げる。 |
| **中規模・複数スケール** | A100 / L40S / 上位 A10 系 | バッチと augment の余裕。 |
| **大規模・蒸留・巨大バッチ** | H100 等 | **単価は高い**ので「本当に必要か」を先に証明。 |

- **複数 GPU 1 台**は **DataParallel** より **`torchrun` + DDP** が主流。  
- **複数ノード**は **NCCL・同一リージョン・近い AZ**、ファイアウォールで **通信ポート**が塞がれていないかを最初に確認。

**落とし穴**: カタログ上の **「GPU メモリ」**だけ見て **CPU メモリ不足**で DataLoader が落ちる、**EBS のスループット不足**で読み込みが律速、などは現場でよくある。

## 4. コストを制御する（プロがやること）

1. **予算アラート・タグ付け**（プロジェクト ID をリソースに付与）。  
2. **Spot / Preemptible / 割安枠** — 学習向き。**中断前提**で `save_every_n_epochs` と **外部ボリューム**にチェックポイント。  
3. **アイドル停止** — インタラクティブ VM は **必ず止める**（スケジュール or ラムダで自動停止）。  
4. **データの置き場** — 何度も S3↔ディスクを往復しない。**学習リージョンとバケットを揃える**。  
5. **過剰プロビジョニング** — 「とりあえず最大 GPU」は **コストとクォータの無駄**。まず **1 GPU で収束確認**。

---

## 5. データとチェックポイント（クラウドで必須の分離）

- **学習データ**: オブジェクトストレージ（S3 / GCS / Azure Blob）＋**マウント or 事前ダウンロード**。  
- **チェックポイント・ログ**: **インスタンス寿命と独立**したボリューム or オブジェクトストレージ。  
- **再開**: `--resume` と **同じ opt の state** まで保存するか方針を決める（検出では **optimizer まで保存しない**運用も多いが、再現性は落ちる）。

**2 分通知で VM が消える**前提で設計する（Spot の典型）。

---

## 6. コンテナと環境の固定

- **PyTorch 公式 Docker** やクラウド提供の **Deep Learning Base AMI / プリビルドコンテナ**を起点にする。  
- 依存は **`pip freeze` ではなく** `requirements.txt` + **ハッシュ** or **`uv.lock`** 等で固定を検討。  
- **CUDA と PyTorch の組み合わせ**は [PyTorch 公式の対応表](https://pytorch.org/get-started/locally/) を参照（クラウドのドライバより **コンテナ側の CUDA** が優先されることが多い）。

---

## 7. 実験管理（クラウドと相性が良い）

- **Weights & Biases**、**MLflow**、**TensorBoard**（GCS/S3 にログ）など、**インスタンス外にメトリクスが残る**ものを併用。  
- **Git コミット hash** と **データセット版（DVC digest 等）**を 1 run に紐づけると、後から「その時の再現」に近づく。

---

## 8. セキュリティ・コンプライアンス（現場で詰まる所）

- **データ持ち込み**: 匿名化・マスキング、**顧客契約でクラウド利用可否**。  
- **ネットワーク**: 学習ノードを **パブリックに晒さない**。**VPC 内のみ**、**S3 VPC endpoint** 等。  
- **シークレット**: API キーは **Secrets Manager** 等、**Git に書かない**。  
- **監査**: 誰がいつどのデータで学習したか **ログを残せるか**（マネージドのジョブ ID が有利なことが多い）。

## 10. アンチパターン（金と時間が消える）

- **チェックポイント無しで Spot**  
- **巨大データを毎回フルダウンロード**（転送課金と律速）  
- **ローカルでは動くがクラウドではパス違い**（データルートをハードコード）  
- **クォータ未確認で複数ノードジョブを投げて全滅**  
- **同じハイパラを闇雲に再実行**（実験管理なし）

## 11. チェックリスト（ジョブ投入前）

- [ ] **データは学習リージョンに近いか**（レイテンシ・転送料）

- [ ] **VRAM に収まる batch / 解像度**を事前に見積もり済み  
- [ ] **予算アラート**と **タグ**  
- [ ] **機密データの持ち込み**が承認済み
