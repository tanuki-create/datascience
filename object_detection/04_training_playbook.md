# 物体検出の学習プレイブック

この文書は、物体検出モデルを「どう学習させると実務で再現性が出るか」を整理する。モデル選定よりも、データ、学習率、解像度、割当、評価、失敗解析の方が結果を左右することが多い。

## まず押さえる全体像

検出モデルの学習は、次の順に考えると安定する。

1. 何を検出したいかを定義する
2. ラベル品質を揃える
3. データ分割で情報漏洩を防ぐ
4. 解像度とバッチサイズをGPU制約内で決める
5. 学習率と損失バランスを合わせる
6. 小物体、不均衡、遮蔽、ドメイン差を潰す
7. 実験を記録し、失敗例を見て修正する

**クラウド GPU** で学習する場合の選定・コスト・Spot・チェックポイントの考え方は [12_cloud_gpu_training_practice.md](./12_cloud_gpu_training_practice.md) を参照。

## データ収集

データは、対象環境、対象カメラ、対象季節、対象業務をできるだけ広く含める。検出は分布シフトに弱いので、理想データだけ集めると本番で落ちる。

- 正例だけでなく、背景や誤検知しやすい負例を集める
- 昼夜、天候、角度、解像度、圧縮劣化を混ぜる
- 似たクラスや紛らわしい物体を意識して入れる

## アノテーション品質

検出では、ラベルのずれがそのまま学習ノイズになる。**クラス設計・版管理・CVAT 運用まで含めたプロ向けの一式**は [11_annotation_guidelines_professional.md](./11_annotation_guidelines_professional.md) を参照。

- bboxは「見える範囲」か「物体の完全外接」かをルール化する
- 遮蔽時の描き方、切れた物体の扱いを統一する
- クラス定義は少なくとも学習前に固定する
- レビューサンプルを作って、ラベルの揺れを定期点検する

## train / val / test 分割

分割はランダムだけでは不十分である。撮影日、現場、個体、カメラ、顧客単位のリークを避ける。

- 同一動画のフレームをtrainとtestに跨がせない
- 同一現場の似た構図を分ける必要があるか検討する
- 長尾クラスはval/testに最低限残す

## アノテーション QA の実務

強いチームは、学習前にラベル品質を可視化します。

- クラス別に 50 件ずつ抜き取りレビューする
- bbox の過小・過大・漏れを分類して記録する
- ラベラー間一致率を定期的に測る
- 曖昧ケースを「規約にない」ではなく、規約更新の材料として残す

ラベル QA をしないままモデル改善を始めると、学習曲線が綺麗でも本番で伸びません。

## 学習率、batch size、image size

これらは独立ではない。実務ではトレードオフとして決める。

- 学習率: 大きすぎるとbboxが暴れ、小さすぎると収束が遅い
- batch size: 大きいほど安定しやすいが、メモリと汎化で妥協が必要
- image size: 小物体が多いほど上げたいが、計算量が増える

よくある方針は、まず標準サイズで走らせ、再現率不足なら解像度を上げ、学習不安定ならLRを下げることである。

## augmentation

augmentationは精度向上だけでなく、分布頑健性を上げる手段である。

- 幾何変換: flip、scale、crop、translate
- 画質変換: blur、noise、JPEG劣化、色変化
- 合成系: mosaic、mixup、copy-paste

注意点は、強すぎるaugmentationでラベル意味を壊さないことである。小物体や密集物体では、過度なcropが有害になりやすい。

## optimizer と学習安定化

多くの検出タスクでは AdamW か SGD が現実的である。

- AdamW: 収束が扱いやすく、Transformer系や微調整で使いやすい
- SGD: 古典的に安定し、CNN系で強いことが多い
- EMA: 推論時の揺れを抑えやすい
- mixed precision: 学習速度とメモリ効率を改善する

## ベースライン設計の原則

最初のベースラインは、最強設定ではなく「壊れたときに原因を追える設定」にします。

- 実績のあるモデルを使う
- 既定の augmentation を大きく崩さない
- seed と実験条件を固定する
- 評価はクラス別、サイズ別、画像付きで残す

最初から複数の改善を同時に入れると、何が効いたか分からなくなります。

## 2024-2026 のコミュニティ実践知

最近の実践知で特に重要なのは、architecture そのものより「再現性のある実装導線」を先に固定することです。

- Ultralytics 系:
  - pretrained モデルから始める
  - YAML で dataset を固定し、`imgsz` と epoch を明示する
  - 最初のベースラインは 640 前後の標準解像度から入る
- MMDetection 系:
  - config を成果物として管理する
  - `auto_scale_lr.base_batch_size` を確認し、batch が変わるなら LR の扱いを明示する
  - AMP、error analysis、confusion matrix を標準作業にする
- RT-DETR 系:
  - custom data 学習、deployment、TensorRT / ONNX / OpenVINO まで含めて評価する
  - small object 案件では sliced inference も検討する

強いチームほど、「どのモデルか」よりも「どの実装・どの設定ファイル・どの重みから始めたか」を厳密に残します。

## freeze / unfreeze と微調整

事前学習バックボーンを使うなら、最初から全部を解凍しない方がよいことが多い。

- まず検出ヘッドだけ学習し、バックボーンは固定する
- その後、上位層から順に解凍する
- DINOv2やViT系では、学習率をバックボーンとヘッドで分けると安定しやすい

## クラス不均衡

物体検出では、背景が圧倒的に多く、さらにクラス頻度も偏る。

- Focal Loss系で易しい負例を抑える
- サンプル再重み付けを使う
- 長尾クラスは収集段階で増やす

## small object 対策

小物体は、解像度不足と特徴解像度不足の両方で落ちる。

- 入力画像サイズを上げる
- FPNや多段特徴を使う
- crop戦略で小物体を相対的に大きくする
- ラベルの粒度を見直す

## 実験管理

検出では、設定のわずかな差が結果に大きく出る。

- データ版数、ラベル版数、seed、解像度、LR、batch sizeを記録する
- mAPだけでなく、precision/recall、クラス別AP、FP/FN例を保存する
- 失敗例を画像で見返せるようにする

## 実験計画の組み方

改善施策は、次の優先度で試すと外れにくいです。

1. ラベル修正
2. データ追加
3. 解像度調整
4. 閾値調整
5. augmentation 調整
6. loss / assignment 調整
7. モデル変更

モデル変更を最後に回すのは、最も複雑で再現性が落ちやすいからです。

## open-vocabulary を使ったデータ立ち上げ

Grounding DINO や YOLO-World のような open-vocabulary detector は、本番 detector の代替より、データ立ち上げの道具として使うと効果が高いです。

実務では次の流れが強いです。

1. promptable detector で候補 bbox を広く拾う
2. 人手で誤検出と class granularity を修正する
3. closed-set detector を fine-tune する
4. 本番では closed-set detector を主系にし、open-vocabulary 系は補助に回す

この流れにすると、long-tail の探索速度と本番安定性を両立しやすくなります。

## foundation backbone を使うときの判断軸

DINOv2 のような backbone を使う場合、次のように判断します。

- ラベル数が少ないか
- ドメイン差が大きいか
- 背景やテクスチャ差が大きいか
- backbone の計算コストを許容できるか

答えが前半で Yes、後半で許容なら検討価値があります。逆に、十分なラベルがあり、速度制約が厳しいなら、まずは成熟した CNN 系 / YOLO 系の強い baseline の方が期待値が高いです。

## デバッグ手順

学習が壊れたら、まずモデルよりデータを見る。

1. 1バッチを可視化する
2. bboxとクラス名の整合性を確認する
3. lossが各項目で発散していないか見る
4. 少数画像で過学習できるか試す
5. 小物体、密集、遮蔽だけのサブセットで評価する

## 推論最適化

本番では、精度だけでなく遅延とスループットを見なければならない。

- batch推論か単発推論かを決める
- NMS閾値とconfidence閾値を業務要件に合わせる
- TensorRT、ONNX、FP16/INT8量子化を検討する
- 入力解像度を下げすぎて再現率を落とさない

## エラー解析

間違いの種類を分けると改善策が見える。

- False Positive: 背景誤検知、類似クラス混同、閾値過敏
- False Negative: 小さすぎる、暗すぎる、遮蔽、ラベル漏れ
- Localization error: bboxがずれる、サイズが合わない

## 本番前チェックリスト

- val 指標だけでなく、擬似本番データで評価したか
- confidence 閾値を業務要件に合わせて決めたか
- 推論時間を本番ハードウェアで測ったか
- 監視する指標とアラート条件を決めたか
- ロールバック手順を用意したか
- モデル更新後に悪化したクラスを検出できる仕組みがあるか

end-to-end / NMS-free 系を本番に出すなら、さらに次も確認します。

- exporter が安定しているか
- profiling が推論環境で再現できるか
- 従来の NMS ベース閾値設計をどう置き換えるか定義したか
- 監視ダッシュボードの指標設計が新しい出力仕様に対応しているか

## 実践的な設定例

### YOLO系ファインチューニングの例

- 入力サイズ: 640 から開始し、小物体が多ければ 960 や 1280 を試す
- batch size: GPUに合わせて最大化するが、学習が不安定なら下げる
- optimizer: まず既定値を使い、必要なら AdamW へ切り替える
- augmentation: mosaic と色変換を有効にし、過剰cropは抑える
- freeze: 先にヘッド中心で数epoch、次に全体微調整
- 評価: mAPだけでなく、見逃し率と誤検知画像を確認する

### ViT / DINOv2 バックボーン活用時の注意点

- 入力解像度を安易に下げない。ViTはパッチ粒度の影響を受けやすい
- バックボーンとヘッドで学習率を分ける
- FPN相当の多段特徴化が必要かを確認する
- 早期に全層を強く更新すると、事前学習表現を壊すことがある
- 小物体が多い場合は、backboneの表現力だけでなく検出頭側の解像度設計が重要である

## この分野の専門家がみな共有している、5つの中核的なメンタルモデル

1. 検出性能はモデル単体ではなく、データ、割当、解像度、後処理を含めた全体で決まる。
2. ラベル品質のばらつきは、そのまま学習ノイズになるので、アノテーション仕様の統一が最優先である。
3. 小物体と長尾クラスは、一般的な平均精度よりも先に落ちる実運用のボトルネックである。
4. 学習の不安定さは、LR、batch、augmentation、freeze/unfreeze の相互作用として起きる。
5. 失敗を画像で見ることが最短経路であり、数値だけでは原因に辿り着けないことが多い。

## この分野の専門家たちが根本的に意見を異にしている点

1. 強い augmentation をかけるべきか
- 強め派の最強主張: 分布変化に強くなり、本番の多様性を先取りできる。
- 控えめ派の最強主張: ラベル意味を壊すと、改善どころか学習信号を損なう。

2. 最初から全層を解凍するべきか
- 全層即解凍派の最強主張: 早くタスク適応でき、最終性能の上限が高いことがある。
- 段階解凍派の最強主張: 事前学習表現を壊しにくく、少データでも安定する。

3. 高解像度に振るべきか、効率を優先すべきか
- 高解像度派の最強主張: 小物体と細部の見逃しが減り、実務精度が上がる。
- 効率派の最強主張: 遅延とコストを守れなければ、精度改善は運用価値を持たない。

## 演習課題

1. 学習が不安定なとき、LR、batch size、augmentation、freeze/unfreeze のどこから疑うか順番を作る
2. ラベル修正よりモデル変更を先にやるべきでない理由を説明する
3. 小物体案件で、解像度向上と crop 戦略のどちらを優先するか議論する
4. DINOv2 バックボーンを使う案件で、全層即解凍が危険な理由を説明する

## セルフチェック

- train / val の差を見て、過学習、リーク、ドメイン差を切り分けられるか
- 実験台帳に最低限何を残すべきか即答できるか
- 推論最適化で削ってはいけない精度を、業務観点で説明できるか
