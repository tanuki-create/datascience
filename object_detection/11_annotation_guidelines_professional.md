# 物体検出アノテーション — プロ向けガイドライン＆運用ドキュメント

この文書は、**バウンディングボックス中心の物体検出**（必要に応じてインスタンス属性）について、**アノテーション設計・作業・品質管理・変更管理**までを一貫して回すための実務標準をまとめたものです。ツールは **CVAT**、**Label Studio** などを想定するが、原則はツール非依存です。

関連: [04_training_playbook.md](./04_training_playbook.md)（学習とデータ品質）、[07_architectural_drawing_symbol_detection.md](./07_architectural_drawing_symbol_detection.md)（図面・高密度記号）、[08_end_to_end_training_workflow.md](./08_end_to_end_training_workflow.md)（ツールスタック）。

---

## 0. 5 つの中核メンタルモデル（プロが先に揃える認識）

1. **アノテは「データ製品」** — 単発の作業ではなく、**定義・版・監査可能な出力**として扱う。  
2. **クラス設計はビジネス境界が主** — 「モデルが分かりそう」より **誤認コスト・運用・KPI** で切る。細かい見た目差は **属性や第 2 段**で吸収しやすい。  
3. **bbox ルールは全員が同じ擬似コードで読める** — 「だいたい囲う」はノイズの源泉。**境界ケースを最初に列挙**する。  
4. **品質は測る** — 抜き取りレビュー、**ラベラー間一致**、エラー分類のログ。感覚だけにしない。  
5. **変更はマイグレーション付き** — クラス増減・統合は **マッピング表と再アノテ範囲**がセット。学習脚本に依存しない。

## 3. クラス設計（taxonomy）— 運用で効くやり方

### 3.1 クラスを増やす前のチェック

- **別クラスにする条件**: 誤認時の**業務影響が違う**、**対応フローが違う**、人間でも**安定的に区別**できる。  
- **属性で足りる条件**: 見た目差だけ／フォーマット版だけ違う／サンプルがまだ薄い → **`subtype` や `schema_rev` 属性**で持つ（後からクラス分割もできる）。  

### 3.2 クラス辞書（Class Registry）に必ず書く項目

- **stable_id**: 学習・評価で使う **不変 ID**（リネームしても ID は変えない）  
- **display_name**: 画面上の表示名（変更可）  
- **定義**: 1〜3 文＋**含む／含まない**  
- **正例・境界例・否例**: 画像またはリンク  
- **親カテゴリ（任意）**: レポート用ロールアップ  
- **priority**: 長尾対策、サンプリング重みの参照用  

### 3.3 スプレッドシート版「クラス × 記号」辞書（推奨）

**結論**: **スプレッドシートで持つ做法はプロでも普通にやる**。図面のように **記号とクラス ID の対応**が多いほど、**CVAT のラベル一覧だけ**だと現場用語と結びつかない。**Class Registry を人間が毎日触る表形式**として一本化し、必要なら §10 の YAML は **CSV からエクスポート**で揃える。

#### 役割

- **単一の正（SoT）** に近づける: 「この記号はどの `stable_id` か」を誰が見ても同じセルを参照する。  
- **アノテ・レビュー・ドメインエキスパート**が同じ言葉で会話できる（略称・JIS・社内記号番号を載せる）。  
- **学習前処理**: `display_name` ↔ `stable_id` のマージや、**対象外クラスのフィルタ**をスプシ列で機械可読にする。

#### 推奨列（図面・設備記号向けにもそのまま使える）

列名はチームで固定し、**先頭行を変更しない**（スクリプトが列名依存になりやすい）。

| 列名（例） | 内容 |
|------------|------|
| `stable_id` | 不変の数値 ID（0 始まりなど学習側に合わせる） |
| `cvat_label` | CVAT / 学習 YAML と**完全一致**する短い英名（空白・全角禁止推奨） |
| `name_ja` | 日本語の正式呼称（レビュー・報告用） |
| `symbol_alias` | 図面・現場での**略号・記号文字**（複数ある場合は `;` 区切り） |
| `doc_ref` | 参照規格・社内図記号表の**版・ページ・URL**（なくても可） |
| `annotate` | `yes` / `no` / `deprecated` — **no は凡例専用・対象外線など** |
| `definition_1line` | 1 行定義（長文は別ドキュメントへリンク） |
| `parent_id` | 親クラス `stable_id`（階層レポート用・空可） |
| `confusable_with` | 紛らわしい **別 `stable_id`**（セミコロン区切り） |
| `bbox_note` | §4 との接続（例: 「外周のみ」「回転記号は軸平行」） |
| `example_url` | 正例画像・社内 Wiki のパス |
| `schema_version` | この行が有効になった **anno-schema**（例 `2.3.0`） |
| `notes` | 改訂理由・チケット ID |

**任意列**: `priority`（サンプル目標）、`hotkey`（CVAT ショートカット）、`train_weight`、`subset`（図種別：電気/空調）。

#### 運用上のコツ

- **権限**: 「誰が編集できるか」を決める。**ラベラー全員が列を増やさない**。  
- **変更**: 行の**削除より `annotate=deprecated` + `legacy_stable_id` 列**で履歴を残すとマイグレが楽。  
- **Git**: 受入れごとに **CSV をエクスポートしてリポジトリにコミット**すると、学習再現と説明責任に効く。  
- **図面**: 記号が増えるたび **辞書を先に更新 → CVAT ラベル追加**の順にすると、**ID ズレ**が減る。

### 3.4 版管理とマッピング

- スキーマに **semantic version**（例: `anno-schema 2.3.0`）を付与。  
- **`legacy_id → current_id`** の対応表を Git や表で保持。学習の前処理で毎回適用。  
- **クラス分割**したら: 影響タスク、**再アノテ範囲**、**既存モデルの扱い**を同じチケットで閉じる。

### 3.5 専門家の間で割れる論点（双方の「最強」主張）

- **主張 A（細かく分ける）**: クラスを細分化すれば **混同行列が読みやすく**、現場の言葉と一致しやすい。  
- **主張 B（粗く保つ）**: クラス数を抑えると **サンプル効率と mAP** が安定し、**長尾地獄**を避けやすい。  

**実務の折衷**: **検出は粗いクラス**、必要なら **属性 or 第 2 分類頭**。KPI だけ細い粒度で切るなら **評価用マッピング**を別に持つ。

## 4. バウンディングボックス規約 — ブレを殺す中心線

### 4.1 基本ルール（どちらか一方を採用し、文書化する）

**方式 A: 完全外接矩形（推奨が多い）**  
- 物体の**見えている全域**を囲む最小の軸平行矩形。  
- **回転物体**でも基本は軸平行（回転 bbox にするかは別決定）。

**方式 B: 可視ピクセル外接**  
- 実際に**不透明に見える部分**のみ。影や反射の扱いを別途定義。

**プロの勘**: 方式を混ぜると **IoU 学習が壊れる**。迷ったら **方式 A** に寄せ、反射・影は「含めない」と明記。

### 4.2 境界ケース一覧（ガイドに必ず入れる）

- **遮蔽**: 見えている部分だけか、全体の推定ボックスか。  
- **画面端で切れている物体**: 切れた部分は**描かないのか**、**推定で全部囲うのか**。  
- **重なり（一般・図面）**: **別インスタンスか同一の塊か**は業務定義で決め、ルールは §4.3 にまとめる。  
- **密集・接触**: **最小の分離可能な個体**単位か。  
- **1 ピクセル級の物体**: **最小 bbox サイズ**（例: 短辺 2px 未満はスキップ）を決めるか、**別タスク**にする。  
- **集合体**（束・パレット）: **1 箱**か **個物**か、業務定義に紐づける。

### 4.3 図面などでの重なり — 異クラス・同クラス・学習側の注意

建築・設備図面などでは物体が重なるのが常態化する。**ラベル規約と後処理（NMS）をセット**で決めないと、検証で初めて「片方の正解が消える」問題が出る。

#### アノテーションの原則

- **論理的に別インスタンス**なら **別 bbox**（**クラスが同じでも同様**。同じ記号が2つ、同型機器が2台 → **2 箱**。重なって **IoU が高くても正解は2箱**になり得る）。  
- **見えている範囲**で規約（§4.1）に沿って囲む。**分析用**に `occluded` などの属性を付けておくと、後から「重なり専用の評価サブセット」を切りやすい。  
- **線で囲まれた1装置**を業務上 **1 数量**と数えるなら **1 bbox**（中の細部の扱いは規約で明示）。  
- **凡例ブロック**: **検出対象外**か **別クラス**かを決め、本体ラベルと混在させない。  
- **完全に隠れて判断不能**なら **bbox なし**が無難。「推定で全体」を許すなら **全員に明文化**した別ルールにする。  

**迷いどころ**は「塊か2つか」だけなので、**数量を数える単位（ビジネス定義）**で一章書いておく。

#### 異クラスが重なる場合

- 意味が別なら **それぞれ bbox**。NMS は通常 **クラス別**なので、**別クラス同士の重なりは NMS で片方が消えにくい**（実装依存のため val で確認）。

#### 同じクラスが重なる場合（特に重要）

- 正解が **意図的に2箱（以上）** の学習データでは、標準的な **NMS** が「同クラス＋高 IoU＝重複」とみなし **どちらかを捨てやすい**。**recall が足りない原因**になり得る。  
- 対応の方向性（いずれも **重なりケースを含む val** で確認）: **NMS IoU 閾値の調整**、**Soft-NMS**、**DETR / NMS-free 系**、**タイル推論後のマージルール**の見直し。  

#### 推論パイプライン（図面では定番）

- 高解像度図面は **オーバーラップ付きタイル**で推論し、境界でインスタンスが切れないようする。詳細は [07_architectural_drawing_symbol_detection.md](./07_architectural_drawing_symbol_detection.md)。  

グレーケースは **Class Registry に境界例画像を追記**し、レビューで逸脱をコード化する。

### 4.4 品質基準（受入れの数値目安の例）

チームで合意したうえで使う（絶対値はドメイン依存）。

- **主要クラス**: ランダム抜取で **bbox IoU（レビューア再描画比）の中央値**がしきい値以上  
- **露見エラー率**: 漏れ・クラス誤り・過大過小をコード別に **周次で推移**  
- **ラベラー間**: 同一画像ダブルアノテで **IoU 分布 or match rate** を四半期で見る

## 5. ツール運用（CVAT を例にしたプロ設定）

### 5.1 プロジェクト設計

- **1 プロジェクト＝1 スキーマ版**（途中でラベル集合が変わるなら **プロジェクトを分ける**か、**タスクを期で分ける**）。  
- **ラベル色**はクラスごと固定。**hotkey** を割り当て、**キーボード中心**で速度と一貫性を上げる。  
- **属性**: `occluded`, `truncated`, `difficult`, `drawing_standard` など。**モデルが使わなくても**分析と再アノテに効く。

### 5.2 ジョブ分割

- **難易度・ドメイン**でバッチを分け、**ゴールデンセット**（検定用固定タスク）を持つ。  
- 新規ラベラーは **キャリブレーション**: ゴールデンで規約理解を確認してから本番。

### 5.3 エクスポート

- **正本（source of truth）**: チームで決める。多くは **CVAT 内の DB + 定期エクスポート**。  
- 学習直前にだけ **COCO / YOLO** に変換するなら、**変換スクリプトの版**も固定（`dataset build vX.Y`）。  
- **画像パス・アノテの hash** をメタデータに残すと、**再現性**が上がる。

### 5.4 Label Studio など他ツールへの注意

- 同じ **Class Registry と bbox 規約**を貼る。  
- バウンディングボックスの **小数・正規化**はエクスポート形式で差が出るため、**訓練コードと突き合わせ**る。

## 6. ワークフロー（推奨フェーズ）

```text
要件定義 → スキーマ v0 → パイロット（少量）→ 規約改訂 → 本番アノテ
    → 継続 QA → スキーマ v1.x → （マイグレーション）→ 学習データ凍結（tag）
```

1. **パイロット**: 100〜300 枚で**混同行・漏れ**の傾向を見てから規約を固める。  
2. **本番**: バッチ単位で **進捗・品質ダッシュボード**（件数、保留件数、エラー率）。  
3. **凍結**: 学習に使う版に **`dataset-2026-03-annotations-2.3.0`** のような **git tag 相当のラベル**を付ける。

## 7. 学習パイプラインとの接続

- **train / val / test** は **現場・日付・センサ・図面プロジェクト**でリークしない（[04](./04_training_playbook.md)）。  
- **長尾クラス**は val/test に最低件数を保証（Registry の priority と連動）。  
- **difficult フラグ**を付けているなら、**学習から外す／重み下げる／別評価**を明示。

---

## 8. 変更管理（インシデントを減らす手順）

1. **変更提案**: 影響（既存枚数、再アノテ工数、モデル互換）。  
2. **承認後**に **スキーマ版を上げる**。  
3. **マッピング表**を更新し、変換スクリプトのテストを追加。  
4. **データセット凍結済み**の実験は **旧版のスナップショット**で再現。

## 11. まとめ

プロとアマの差は、**綺麗な矩形を引けるか**ではなく、**(1) 変わっても壊れない定義と版 (2) 測れる品質 (3) 学習と矛盾しない bbox 規約** を運用に落とせるかにあります。この文書を **自チーム向けに短くカスタム**し、**Class Registry とセット**でリポジトリまたは Wiki のトップに置くのがよい構成です。
